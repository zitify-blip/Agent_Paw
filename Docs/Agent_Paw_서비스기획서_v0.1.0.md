## 1. 서비스 개요

### 1.1 한 줄 요약

Agent Paw는 소프트웨어 개발팀이 역할별 AI 에이전트(페르소나)와 협업하여 설계·개발·문서화 작업을 수행하는 멀티 에이전트 오케스트레이션 데스크톱 플랫폼이다.

### 1.2 서비스 배경

. 단일 AI 대화 도구에 의존하는 팀은 역할 혼재, 컨텍스트 유실, 결정 이력 부재 문제를 반복적으로 경험한다.
. PM·개발자·QA·DA 등 역할별 전문 지식을 보유한 복수의 AI 에이전트가 동일 프로젝트 컨텍스트를 공유하며 협업하면, 더 정확하고 실행 가능한 산출물을 생성할 수 있다.
. 사내 보안 요건상 소스코드·설계서를 외부 SaaS 서비스에 업로드하기 어려운 팀을 위해 로컬 설치형(on-premise) 방식으로 제공한다.

### 1.3 핵심 가치

[User·팀·기술 세 관점에서 Agent Paw가 제공하는 가치가 구분된다.]

| 관점 | 가치 |
|---|---|
| User | 반복적인 문서 작업과 코드 리뷰를 AI 에이전트에게 위임하여 핵심 업무에 집중한다. |
| 팀 | 대화·결정·산출물 이력이 프로젝트 단위로 자동 기록되어 온보딩·지식 공유 비용이 감소한다. |
| 기술 | 모든 데이터를 로컬 PostgreSQL에만 저장하므로 코드·설계 데이터가 외부로 유출되지 않는다. |

### 1.4 서비스 구성

Agent Paw는 두 개의 독립된 프로그램으로 구성된다. 버전은 각각 독립적으로 관리한다.

| 구성 요소 | 플랫폼 | 주요 역할 |
|---|---|---|
| PC 앱 | Windows (WPF, .NET 10) | AI 에이전트 오케스트레이션, 프로젝트·페르소나·산출물 관리 |
| 모바일 앱 | Android (Flutter) | PC 앱에 원격 접속하여 프로젝트 조회·대화 수행 |

단, 모바일 앱은 PC 앱의 MobileApi 엔드포인트(포트 47893)에 직접 연결한다. PC 앱이 실행 중이어야 모바일 앱이 동작한다.

## 2. 용어 정리

| 용어 | 정의 |
|---|---|
| 페르소나(Persona) | AI 에이전트의 역할 정체성. 시스템 프롬프트·언어 모델·온도 설정값을 포함한다. |
| PM 페르소나 | IsPm=true인 특수 페르소나. 토론 조율·종료 보고·User 개입 요청 권한을 단독으로 가진다. |
| 지침(Instruction) | 페르소나 또는 프로젝트에 연결하는 Markdown 텍스트 파일. 시스템 프롬프트에 주입된다. |
| 워크스페이스(Workspace) | 프로젝트 단위의 로컬 파일 경로. AI 에이전트의 파일 I/O 작업 범위를 한정한다. |
| 오케스트레이터(Orchestrator) | 입력 메시지를 분석하여 적절한 에이전트 파이프라인을 선택·실행하는 핵심 서비스. |
| 핸드오프(Handoff) | 응답 중인 에이전트가 다음 에이전트를 지명하여 대화를 이전하는 행위. |
| 스냅샷(Snapshot) | 특정 시점의 프로젝트 상태를 저장한 기록. Git 커밋 SHA와 연결된다. |
| 롤백(Rollback) | 스냅샷 시점으로 프로젝트 상태를 복원하는 작업. |
| 위키(Wiki) | AI 에이전트가 자동 또는 User가 수동으로 생성한 프로젝트 지식 문서. ADR·SPEC·TROUBLE 세 카테고리로 분류된다. |
| 팀 모드 | User가 에이전트 2명 이상을 선택하여 실행하는 협업 대화 모드. Panel·Debate·Chain 세 가지를 제공한다. |
| 자유 토론 모드 | PM 페르소나와 비PM 에이전트가 혼재한 프로젝트에서 System이 자동으로 진입하는 다자 토론 파이프라인. |
| 이벤트 로그(EventLog) | 모든 대화 턴의 입력·출력·도구 호출 이력을 JSON으로 영구 저장하는 감사 기록. |
| MobileApi | 모바일 앱이 PC 앱에 접속하기 위한 로컬 HTTP 서버. 포트 47893을 사용한다. |
| RunId | 단일 파이프라인 실행을 식별하는 8자 무작위 ID. |

## 3. 타겟 사용자

### 3.1 사용자 페르소나

**페르소나 A: 기획자·PM (주 사용자)**

| 항목 | 내용 |
|---|---|
| 역할 | 서비스 기획, 요구사항 정의, 문서 산출 |
| 주요 목적 | AI 에이전트와 대화하여 서비스 정책·기술 명세·사용자 스토리를 초안 생성하고 검토한다. |
| 핵심 니즈 | 반복 문서 작업 자동화, 의사결정 이력 보존, 산출물 버전 관리 |
| 사용 시나리오 | "새 기능의 FSD를 작성해줘" → AI 에이전트가 구체적 명세를 생성 → 위키에 자동 저장 |

**페르소나 B: 개발자 (보조 사용자)**

| 항목 | 내용 |
|---|---|
| 역할 | 기능 구현, 코드 리뷰, 기술 문서 작성 |
| 주요 목적 | 코드 분석·리팩토링·테스트 케이스 생성을 AI에게 위임한다. 파일 첨부 또는 파일 I/O 도구로 로컬 코드를 직접 참조시킨다. |
| 핵심 니즈 | 로컬 코드베이스 직접 접근, 코드·설계 외부 유출 차단, 빠른 컨텍스트 전환 |
| 사용 시나리오 | 특정 `.md` 파일을 첨부하여 "이 명세 기반으로 테스트 케이스 작성해줘" 요청 → AI가 명세를 읽고 케이스 파일 생성 |

**페르소나 C: 팀 리드 (관리자)**

| 항목 | 내용 |
|---|---|
| 역할 | 기술 의사결정, 팀 표준 수립, AI 에이전트 구성 관리 |
| 주요 목적 | 프로젝트별 페르소나·지침을 구성하여 팀 전체가 일관된 AI 보조를 받도록 환경을 설정한다. |
| 핵심 니즈 | 페르소나 커스터마이징, API 키 중앙 관리, 챗봇 플랫폼 연동 |
| 사용 시나리오 | 신규 프로젝트 생성 → 역할별 페르소나 구성 → 지침 파일 연결 → 챗봇 채널 링크 |

## 4. 서비스 정책

### 4.1 인증 정책

인증은 Google OAuth 2.0을 기본 수단으로 한다. 로컬 개발 환경을 위한 Dev Bypass 모드를 병행 제공한다.

| 항목 | 정책 |
|---|---|
| 인증 방식 | Google OAuth 2.0. System이 Access Token과 Refresh Token을 발급·관리한다. |
| 세션 유지 | System이 Refresh Token을 AES-256으로 암호화하여 로컬 DB에 저장한다. 앱 재시작 시 System이 자동 복원을 시도한다. |
| Dev Bypass | `appsettings.json`에 `MobileApi.DevUserId` 값이 존재하는 경우에만 Dev Bypass 버튼을 노출한다. 프로덕션 환경에서는 해당 키를 제거한다. |
| 로그아웃 | User가 로그아웃하면 System이 로컬 세션 토큰을 즉시 파기하고 UI를 로그인 화면으로 전환한다. |
| Google Docs 인가 | Google Docs 내보내기 사용 시 User가 Google 계정으로 추가 인가를 수행한다. 별도 Refresh Token을 발급·저장한다. |

단, Google OAuth 리다이렉트 포트는 47891이다. 해당 포트에 방화벽 인바운드 허용 규칙이 선행되어야 한다.

### 4.2 프로젝트 정책

| 항목 | 정책 |
|---|---|
| 생성 | User가 이름·설명을 입력하여 프로젝트를 생성한다. ProjectId는 System이 GUID로 자동 채번한다. |
| 계층 구조 | 프로젝트는 ROOT / PARENT / CHILD 3계층 구조를 지원한다. |
| 보관(Archive) | User가 보관하면 대시보드 기본 목록에서 숨긴다. 보관 해제 시 복원한다. 보관 중인 프로젝트는 AI 파이프라인 실행이 불가하다. |
| 삭제 | User가 삭제하면 System이 Status=DELETED로 논리 삭제한다. 물리 삭제 시점은 별도 운영 정책으로 결정한다. |
| 워크스페이스 경로 | 기본값은 `%LOCALAPPDATA%\AgentPaw\projects\{ProjectId}\workspace`이다. User가 임의 경로로 변경 가능하다. |
| Git 연동 | User가 외부 레포를 클론하거나 워크스페이스에 Git 레포를 직접 초기화할 수 있다. PAT(Personal Access Token)는 UI 세션에서만 임시 보관하며 DB에 저장하지 않는다. |

### 4.3 페르소나·지침 정책

| 항목 | 정책 |
|---|---|
| 글로벌 페르소나 | System이 앱 최초 설치 시 기본 글로벌 페르소나 템플릿(PM, DevAgent, QA, DA 등)을 자동 시드한다. ProjectId=null로 구분한다. |
| 프로젝트 페르소나 | User가 글로벌 페르소나를 프로젝트에 링크하거나 프로젝트 전용 페르소나를 직접 생성한다. |
| PM 지정 | 프로젝트당 PM 페르소나는 1명을 권장한다. PM이 없으면 자유 토론 파이프라인은 비활성화된다. |
| 지침 파일 형식 | Markdown 텍스트 파일이다. LLM 컨텍스트 창 내에서 수용 가능한 크기를 권장한다. |
| 지침 주입 순서 | 프로젝트 수준 지침이 선주입되고 페르소나 수준 지침이 후주입된다. 동일 파일의 중복 주입은 불가하다. |
| 삭제 | 페르소나 삭제 시 해당 페르소나의 이벤트 로그는 보존한다. 논리 삭제 방식을 적용한다. |

### 4.4 AI 파이프라인 정책

| 항목 | 정책 |
|---|---|
| 단일 에이전트 | 기본 파이프라인. PM이 없거나 @멘션으로 특정 에이전트를 지정한 경우에 동작한다. |
| 자유 토론 | PM과 비PM 에이전트가 각 1명 이상 있는 프로젝트에서 System이 자동으로 진입한다. |
| 팀 모드 | User가 에이전트 2명 이상을 선택한 경우에 동작한다. Panel / Debate / Chain 세 가지 모드를 제공한다. |
| 최대 반복 한계 | 단일 파이프라인 최대 24턴, 다자 토론 최대 10라운드(참여자 최대 4명), Handoff 최대 12회. 한계 도달 시 System이 경고 메시지를 노출하고 파이프라인을 종료한다. |
| 토론 중단 | User가 "중단" 버튼을 클릭하면 System이 현재 실행 중인 파이프라인에 취소 신호를 전달한다. 완성된 턴의 이벤트 로그는 보존한다. 진행 중이던 스트리밍 버퍼는 소멸한다. |
| 스트리밍 | 모든 AI 응답은 토큰 단위로 실시간 스트리밍된다. UI는 33ms(≈30fps) 간격으로 갱신한다. |
| 자동 위키 저장 | 에이전트가 `wiki_save` 블록을 포함한 응답을 반환하면 System이 자동으로 위키 문서를 생성·갱신한다. 동일 ProjectId + Category + Title 문서가 존재하면 System이 내용을 덮어쓰기한다. |
| User 개입 요청 | PM이 `pm_intervention` 블록을 반환하면 System이 대화를 일시 정지하고 User에게 추가 입력을 요청한다. |
| "사용자에게 묻기" | 워크스페이스 상단 체크박스로 제어한다. 해제 시 PM이 User에게 되묻는 행위가 완전히 금지된다. |
| 모델 설정 | 페르소나별 PrimaryModel과 FallbackModel을 지정한다. API 과부하 시 System이 FallbackModel로 자동 전환한다. |

**팀 모드 동작 정의**

| 모드 | 동작 |
|---|---|
| Panel | 선택된 에이전트가 순서대로 독립적으로 응답한다. |
| Debate | 에이전트들이 라운드 테이블 방식으로 토론한다. 모든 참여자가 "agree" 입장이면 조기 종료한다. |
| Chain | 앞 에이전트의 출력이 다음 에이전트의 입력으로 연결된다. |

### 4.5 데이터 보존 정책

| 데이터 | 저장 위치 | 보존 기간 |
|---|---|---|
| 이벤트 로그 | PostgreSQL EventLog 테이블 | 삭제 요청 시까지 영구 보존 |
| 위키 문서 | PostgreSQL WikiDocument 테이블 | 삭제 요청 시까지 영구 보존 |
| 스냅샷 | PostgreSQL Snapshot 테이블 + Git | 롤백·삭제 시까지 보존 |
| AI 응답 스트리밍 버퍼 | 메모리 (런타임) | 세션 종료 시 소멸 |
| API 키 | PostgreSQL ApiKeyStore 테이블 (AES-256 암호화) | 삭제 요청 시까지 영구 보존 |
| 인증 토큰 | PostgreSQL AuthToken 테이블 (AES-256 암호화) | 로그아웃 시 즉시 파기 |
| PAT | 메모리 (UI 세션) | 앱 종료 시 소멸. DB에 저장하지 않는다. |

### 4.6 외부 연동 정책

| 연동 서비스 | 활성화 조건 | 비고 |
|---|---|---|
| Google Chat | GCP 프로젝트 ID + Pub/Sub 구독 ID + 서비스 계정 JSON 모두 설정 | System이 Pub/Sub Pull 방식으로 메시지를 수신한다. |
| Slack | Bot Token + App Token(Socket Mode) 모두 설정 | System이 WebSocket 기반으로 실시간 메시지를 수신한다. |
| Telegram | Bot Token 설정 | System이 폴링 방식으로 메시지를 수신한다. |
| WebSocket 서버 | 설정 화면에서 활성화 | 외부 시스템이 빌드 결과·AI 알림을 수신하는 단방향 서버 역할을 한다. |
| MobileApi | DevToken + DevUserId 모두 설정 | 포트 47893. 방화벽 인바운드 규칙 설정이 선행되어야 한다. |

단, 챗봇 플랫폼은 프로젝트 단위로 Space·채널을 링크한 경우에만 AI 파이프라인을 실행한다. 링크 없는 Space에서 수신된 메시지는 System이 무시한다.

## 6. 외부 연동 명세

### 6.1 챗봇 플랫폼 처리 흐름

| 단계 | 트리거 | Server 동작 | DB / 외부 연동 |
|---|---|---|---|
| 1. 메시지 수신 | 외부 플랫폼 메시지 도착 | 플랫폼별 수신 서비스(PubSub Pull / Slack Socket / Telegram Polling)가 메시지를 수신한다. | — |
| 2. 라우팅 | — | ChatDispatcherService가 Space·채널 링크를 조회하여 연결된 ProjectId를 확인한다. 링크 없으면 무시한다. | ChatSpaceLink SELECT |
| 3. 파이프라인 실행 | — | OrchestratorService.RunPipelineAsync를 호출한다. | EventLog INSERT |
| 4. 응답 전송 | — | 플랫폼별 SendMessageAsync로 AI 응답을 전송한다. | — |

단, Google Chat Pub/Sub는 At-least-once 수신 보장 정책을 사용한다. 메시지 중복 수신 시 동일 RunId가 생성될 수 있다. 멱등성 보장은 현재 미구현이며 백로그로 관리한다.

### 6.2 모바일 앱 연동

| 항목 | 내용 |
|---|---|
| 서버 주소 | PC 앱이 실행 중인 로컬 IP:47893 |
| 인증 방식 | Dev 토큰 (`appsettings.json`의 `MobileApi.DevToken`) |
| 주요 엔드포인트 | `/m/health` (연결 확인), `/m/me` (인증 확인), `/m/projects` (프로젝트 목록), `/m/chat` (대화) |
| 방화벽 | 인바운드 TCP 47893 허용 규칙이 선행되어야 한다. Inno Setup 설치 시 System이 자동 등록한다. |

단, `DevToken`과 `DevUserId`가 모두 설정된 경우에만 System이 MobileApi 서버를 시작한다. 어느 하나라도 미설정 시 서버가 시작되지 않으므로 모바일 앱 연결이 불가하다.

## 7. 비기능 요구사항

### 7.1 보안

| 항목 | 정책 |
|---|---|
| API 키·인증 토큰 암호화 | AES-256으로 암호화하여 DB에 저장한다. 평문 저장은 불가하다. |

| 항목 | 목표 |
|---|---|

| 항목 | 정책 |
|---|---|

## 예외 처리 및 제약사항

. AI 응답 중 네트워크 오류 발생 시 System이 오류 메시지를 error 역할로 채팅 목록에 노출하고 `IsLoading=false`로 전환한다.
. 파이프라인 취소(토론 중단) 시 진행 중이던 스트리밍 버퍼는 소멸한다. 이미 완성된 턴의 이벤트 로그는 보존한다.
. 최대 반복 횟수 도달 시 System이 "⚠ 반복 상한(24회)에 도달해 작업을 중단합니다." 메시지를 노출하고 파이프라인을 종료한다.
. Git 클론 중 PAT 오류 발생 시 System이 오류 메시지를 CloneMessage에 노출하고 IsCloning을 false로 전환한다.

단, PostgreSQL이 미실행 상태에서 앱을 시작하면 DB 연결 오류가 발생한다. System이 오류 팝업을 노출하고 재시작을 안내한다.
※ appsettings.json 미설정 시 MobileApi 서버가 시작되지 않는다. 모바일 앱 연결 불가 상태는 별도 로그로 확인한다.

본 문서는 애플리케이션 개발 전 과정에서 준수해야 하는 보안 규칙을 정의한다. SQL Injection·XSS·CSRF·인증/인가·암호화의 기본 원칙을 다룬다.

---

## 1. 입력 검증 원칙

. 모든 외부 입력(요청 파라미터·헤더·쿠키·파일·환경변수)을 신뢰하지 않는다.
. 화이트리스트 방식으로 검증한다. 블랙리스트 방식을 지양한다.
. 검증 실패 시 상세 원인을 응답에 노출하는 것을 금지한다. 일반화된 메시지로 응답한다.

## 2. SQL Injection 방어

. 모든 쿼리는 파라미터화 쿼리(Prepared Statement) 를 사용한다.
. 문자열 연결·템플릿 리터럴을 통한 쿼리 조립을 절대 금지한다.
. ORM 사용 시에도 Raw Query는 반드시 파라미터 바인딩을 사용한다.

```
❌  db.query(`SELECT * FROM user WHERE id = '${userId}'`)
✅  db.query('SELECT * FROM user WHERE id = ?', [userId])
```

. 테이블명·컬럼명을 동적으로 주입해야 하는 경우 화이트리스트 검증을 선행한다.

## 3. XSS 방어

| 방어 계층 | 조치 |
|---|---|
| 출력 인코딩 | HTML·JS·URL 컨텍스트별 적절한 이스케이프를 수행한다 |
| CSP 헤더 | `Content-Security-Policy` 를 엄격 모드로 설정한다 |
| HttpOnly 쿠키 | 세션 쿠키에 `HttpOnly`, `Secure`, `SameSite=Strict` 속성을 부여한다 |
| innerHTML 금지 | DOM 조작 시 `textContent` 를 사용한다. `innerHTML` 사용을 제한한다 |
| 템플릿 엔진 | 자동 이스케이프를 활성화한다. Raw 출력(`{{{ }}}`)을 금지한다 |

## 4. CSRF 방어

. 상태 변경 요청(POST/PUT/DELETE)에 CSRF 토큰을 필수 적용한다.
. SameSite 쿠키 속성을 `Strict` 또는 `Lax` 로 설정한다.
. Origin·Referer 헤더 검증을 병행한다.
. 중요 동작(결제·비밀번호 변경)은 재인증을 요구한다.

## 5. 인증·인가

### 5.1. 인증(Authentication)

. 비밀번호 최소 요구사항 : 12자 이상, 영문·숫자·특수문자 각 1자 이상 포함.
. 비밀번호 해싱 : `bcrypt`, `argon2id`, `scrypt` 중 택일한다. MD5·SHA-1 사용을 절대 금지한다.
. 로그인 실패 5회 누적 시 System이 계정을 30분간 잠금 처리한다.
. MFA(다단계 인증) 를 관리자 계정에 필수 적용한다.

### 5.2. 인가(Authorization)

. 인가 검증은 반드시 서버 측에서 수행한다. 클라이언트 검증은 UX 보조 용도이다.
. RBAC(Role-Based Access Control) 또는 ABAC 를 사용한다.
. 모든 API 엔드포인트에 인가 미들웨어를 적용한다. 기본은 Deny, 명시적 Allow 방식이다.
. IDOR(Insecure Direct Object Reference) 방어 : 자원 ID 접근 시 소유권을 검증한다.

### 5.3. 토큰

. JWT 사용 시 서명 알고리즘을 `HS256` 또는 `RS256` 으로 고정한다. `none` 허용을 절대 금지한다.
. Access Token 유효기간 : 15분 이내.
. Refresh Token 유효기간 : 14일 이내. DB에 저장하여 무효화 가능하도록 관리한다.
. 토큰을 localStorage 에 저장하는 것을 지양한다. HttpOnly 쿠키를 권장한다.

## 6. 암호화

| 대상 | 알고리즘 | 비고 |
|---|---|---|
| 전송 구간 | TLS 1.2 이상 | TLS 1.0 / 1.1 사용 금지 |
| 저장 구간 (대칭) | AES-256-GCM | ECB 모드 사용 금지 |
| 저장 구간 (비대칭) | RSA 2048 이상 또는 ECDSA P-256 | |
| 해시 (비밀번호) | argon2id / bcrypt | |
| 해시 (무결성) | SHA-256 이상 | MD5·SHA-1 사용 금지 |
| 난수 생성 | CSPRNG (`crypto.randomBytes` 등) | `Math.random()` 사용 금지 |

. 암호화 키는 Secret Manager에서 관리한다. 코드·설정 파일 포함을 금지한다.
. 암호화 키 로테이션 주기 : 최소 1년.

## 7. 보안 헤더

| 헤더 | 권장 값 |
|---|---|
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` |
| `Content-Security-Policy` | `default-src 'self'` (엄격 모드) |
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | 사용 기능만 명시적 허용 |

## 8. 파일 업로드

. 파일 확장자·MIME 타입·매직 넘버(Magic Number) 3중 검증을 수행한다.
. 업로드 파일은 웹 루트 외부 또는 별도 스토리지(S3 등)에 저장한다.
. 저장 파일명은 System이 재생성한다. 원본 파일명을 그대로 사용하는 것을 금지한다.
. 파일 크기 상한을 강제한다. 기본 10MB 이하.

## 9. 로그·응답에서의 정보 노출 금지

. 스택 트레이스·SQL 쿼리·내부 경로를 사용자 응답에 포함하는 것을 절대 금지한다.
. 에러 응답은 `trace_id` 만 노출한다. 내부 정보는 서버 로그에서만 조회한다.
. 민감정보 마스킹은 `개발/로깅규칙.md §4` 를 참조한다.

## 10. 취약점 대응

. 신고 접수 : `security@[domain]` 전용 창구를 운영한다.
. 대응 기한 : `개발/의존성관리.md §3.3` 대응 우선순위표를 따른다.
. 사후 검토 : 보안 사고 발생 시 Post-Mortem 문서를 14일 이내 작성한다.

단, 긴급 보안 패치는 정상 릴리즈 프로세스를 생략하고 hotfix로 배포한다.

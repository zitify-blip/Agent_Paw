본 문서는 Conventional Commits 규약 기반의 커밋 메시지 작성 규칙을 정의한다. 일관된 커밋 메시지는 CHANGELOG 자동 생성과 이력 추적을 가능하게 한다.

---

## 1. 메시지 구조

```
<type>(<scope>): <subject>

<body>

<footer>
```

. 헤더(type·scope·subject)는 필수이다.
. body·footer는 선택이다. 단, 주요 변경(Breaking Change)은 footer에 `BREAKING CHANGE:` 를 명시한다.

## 2. 커밋 타입

| 타입 | 용도 | 예시 |
|---|---|---|
| `feat` | 신규 기능 추가 | `feat(auth): 소셜 로그인 지원 추가` |
| `fix` | 버그 수정 | `fix(order): 결제 금액 반올림 오류 수정` |
| `docs` | 문서 수정 | `docs(readme): 설치 절차 보강` |
| `style` | 포매팅·세미콜론 등 (로직 변경 없음) | `style(user): 들여쓰기 정리` |
| `refactor` | 리팩토링 (기능 변경 없음) | `refactor(payment): Strategy 패턴 적용` |
| `perf` | 성능 개선 | `perf(list): N+1 쿼리 제거` |
| `test` | 테스트 추가·수정 | `test(auth): 로그인 실패 케이스 추가` |
| `chore` | 빌드·패키지·기타 잡무 | `chore(deps): lodash 4.17.21 갱신` |
| `ci` | CI 설정 변경 | `ci: GitHub Actions 캐시 추가` |
| `build` | 빌드 시스템 변경 | `build: Webpack 5로 업그레이드` |
| `revert` | 이전 커밋 되돌림 | `revert: feat(auth) 소셜 로그인` |

## 3. 작성 규칙

### 3.1. subject (제목)

. 50자를 초과하지 않는다.
. 한글 작성을 허용한다. 영어 작성 시 명령문·현재형을 사용한다.
. 마침표(`.`)를 붙이지 않는다.
. 첫 글자는 소문자로 시작한다(영문 한정).

```
❌  feat: 로그인 기능을 추가했습니다.
❌  feat: Added login feature.
✅  feat(auth): 로그인 기능 추가
✅  feat(auth): add login feature
```

### 3.2. scope (범위)

. 변경 범위를 모듈·컴포넌트 단위로 소문자·하이픈으로 표기한다.
. 전체 영향 시 scope를 생략한다.

### 3.3. body (본문)

. subject와 1행을 띄우고 작성한다.
. 72자마다 개행한다.
. "무엇을(What)"과 "왜(Why)"를 설명한다. "어떻게(How)"는 코드에 맡긴다.
. 한글 작성을 허용한다.

### 3.4. footer (꼬리)

. 이슈 참조 : `Refs: APP-123`, `Closes: #456`
. 주요 변경 : `BREAKING CHANGE: [변경 내용 및 마이그레이션 안내]`
. 공저자 : `Co-Authored-By: 이름 <email>`

## 4. 완성 예시

```
feat(payment): 카카오페이 결제 수단 추가

- 카카오페이 SDK v2.3.0 연동
- 결제 완료 콜백 처리 로직 추가
- 결제 실패 시 재시도 UI 노출

기존 결제 수단(신용카드·계좌이체)에 영향 없음.

Refs: APP-201
```

## 5. 금지 사항

. 하나의 커밋에 여러 타입을 혼합하는 것을 금지한다. 분리한다.
. 의미 없는 메시지(`wip`, `fix`, `update`, `임시`)를 금지한다.
. 100줄 이상의 변경을 단일 커밋으로 묶는 것을 지양한다.
. 민감정보(비밀번호·API 키·토큰)를 메시지에 포함하는 것을 절대 금지한다.

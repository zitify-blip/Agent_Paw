본 문서는 Git Flow 기반의 브랜치 운영 규칙을 정의한다. 모든 개발자는 본 전략에 따라 브랜치를 생성·병합·삭제한다.

---

## 1. 브랜치 종류

| 브랜치 | 용도 | 수명 | 병합 대상 |
|---|---|---|---|
| `main` | 프로덕션 배포 | 영구 | - |
| `develop` | 다음 릴리즈 통합 | 영구 | `main` |
| `feature/*` | 신규 기능 개발 | 일시 | `develop` |
| `release/*` | 릴리즈 준비 | 일시 | `main`, `develop` |
| `hotfix/*` | 프로덕션 긴급 수정 | 일시 | `main`, `develop` |
| `bugfix/*` | develop 버그 수정 | 일시 | `develop` |

## 2. 브랜치명 규칙

. 형식 : `[유형]/[이슈번호]-[짧은설명]`
. 소문자·하이픈(`-`)만 사용한다. 공백·대문자·한글을 금지한다.
. 이슈 번호는 이슈 트래커(Jira/GitHub Issues) 번호를 참조한다.

```
❌  feature/새기능 추가
❌  Feature/NEW_LOGIN
✅  feature/APP-123-add-login
✅  hotfix/APP-456-fix-payment-timeout
✅  bugfix/APP-789-resolve-null-pointer
```

## 3. 워크플로우

### 3.1. 기능 개발 흐름

1. `develop` 에서 `feature/*` 브랜치를 분기한다.
2. 개발자는 feature 브랜치에서 커밋한다.
3. 완료 시 `develop` 으로 PR을 생성한다.
4. 리뷰·승인 후 `develop` 에 Squash Merge 한다.
5. 병합 후 feature 브랜치를 삭제한다.

### 3.2. 릴리즈 흐름

1. `develop` 에서 `release/v[버전]` 브랜치를 분기한다.
2. QA·버그 수정은 release 브랜치에서만 수행한다.
3. 릴리즈 확정 시 `main` 과 `develop` 에 병합한다.
4. `main` 에 버전 태그(`v1.0.0`)를 생성한다.

### 3.3. 핫픽스 흐름

1. `main` 에서 `hotfix/*` 브랜치를 분기한다.
2. 긴급 수정 후 `main` 과 `develop` 양쪽에 병합한다.
3. 패치 버전 태그(`v1.0.1`)를 생성한다.

## 4. 보호 규칙

. `main` / `develop` 브랜치에 직접 push를 불가하다. 반드시 PR을 경유한다.
. `main` 병합은 최소 2인의 리뷰 승인을 요구한다.
. `develop` 병합은 최소 1인의 리뷰 승인을 요구한다.
. Force push(`git push -f`)를 보호 브랜치에 금지한다.
. 병합 전 CI 통과(빌드·테스트·린트)를 필수로 한다.

## 5. 병합 전략

| 대상 | 전략 | 사유 |
|---|---|---|
| `feature/*` → `develop` | Squash Merge | 커밋 이력 정리 |
| `release/*` → `main` | Merge Commit | 릴리즈 시점 보존 |
| `hotfix/*` → `main` | Merge Commit | 핫픽스 이력 보존 |
| `main` → `develop` | Merge Commit | 양방향 동기화 |

## 6. 브랜치 생명주기

. feature·bugfix 브랜치는 병합 직후 삭제한다.
. release·hotfix 브랜치는 태그 생성 후 삭제한다.
. 장기 미병합 브랜치(30일 초과)는 System이 주간 알림을 발송한다.

단, WIP 장기 브랜치는 매주 `develop` 을 rebase·merge하여 충돌을 최소화한다.

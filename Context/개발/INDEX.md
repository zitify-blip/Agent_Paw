## 1. 가이드 목록

### 1.1. 코드 품질

. `코딩컨벤션.md` : 들여쓰기·공백·괄호·한 줄 길이 등 공통 코딩 스타일 규칙
. `네이밍규칙.md` : 변수·함수·클래스·상수·파일 네이밍 컨벤션 (camelCase / PascalCase / SNAKE_CASE)
. `디렉토리구조.md` : 프로젝트 루트 디렉토리 배치 및 모듈 구성 원칙

### 1.2. 협업 프로세스

. `브랜치전략.md` : Git Flow 기반 브랜치 운영 규칙 (main / develop / feature / hotfix)
. `커밋메시지.md` : Conventional Commits 기반 커밋 메시지 작성 규칙
. `PR규칙.md` : Pull Request 제목·본문·크기·체크리스트 규칙
. `코드리뷰.md` : 리뷰어 체크리스트, 승인 기준, 응답 시간 규칙

### 1.3. 런타임 품질

. `로깅규칙.md` : 로그 레벨·포맷·민감정보 마스킹 규칙
. `에러처리.md` : 예외 클래스 계층, try-catch 원칙, 에러 코드 체계
. `테스트작성.md` : 단위·통합·E2E 테스트 구분, 커버리지 기준, 테스트명 규칙

### 1.4. 운영·보안

. `의존성관리.md` : 패키지 추가 승인 절차, 버전 고정, 취약점 점검
. `환경변수.md` : `.env` 관리, 시크릿 분리, 환경별 분기(dev/stg/prd)
. `보안지침.md` : SQL Injection·XSS·CSRF·인증/인가·암호화 규칙
. `배포전략.md` : GitHub Releases + Inno Setup + SHA256 기반 자동 업데이트 파이프라인, 버전 동기화 4곳, SemVer 적용 원칙, 긴급 롤백 절차

## 2. 가이드 간 우선순위

. CLAUDE.md 가 최상위 권위 문서이다. 충돌 시 CLAUDE.md 를 따른다.
. 개발 가이드 간 충돌이 발생하면 `보안지침.md` > `에러처리.md` > `코딩컨벤션.md` 순으로 우선한다.

단, 언어·프레임워크 공식 가이드(ESLint, PEP 8 등)와 충돌하는 경우 본 가이드를 우선 적용한다.

## 3. 관점 간 경계

타 관점과 접점이 있는 파일은 상대 문서를 명시적으로 참조한다.

| 개발 측 파일 | 상대 관점 파일 | 경계 내용 |
|---|---|---|
| `테스트작성.md` | `QA/테스트레벨.md` | 단위·통합 테스트는 개발 책임, System·UAT는 QA 책임 |
| `에러처리.md` | `디자인/상태표현.md` | 에러 상태 UI 표현은 디자인, 에러 코드·예외 처리 구현은 개발 |
| `네이밍규칙.md` | `DA/테이블네이밍.md`, `DA/컬럼네이밍.md` | 변수·함수 네이밍은 개발, DB 객체 네이밍은 DA |
| `PR규칙.md`·`코드리뷰.md` | `디자인/핸드오프.md` | 디자인 핸드오프 수령 후 개발 PR 단계 진입 |
| `보안지침.md` | `DA/마이그레이션규칙.md` | 시크릿·암호화는 개발, 스키마 마이그레이션 절차는 DA |

## 4. 자가점검 적용

. 개발 산출물 변경 시 `Context/자가점검_워크플로.md` 의 M01·M02·M17·M18(코드 스타일·린트) 을 필수로 적용한다.
. 테스트 변경 동반 시 M20(테스트 커버리지·레벨) 을 추가한다.
. 2개 이상 관점 동시 변경 시 M22(관점 간 경계) 를 1회 이상 포함한다.

## 1. 배포 파이프라인 개요

AgentPaw 는 **GitHub Releases 기반 자동 업데이트** 방식으로 배포한다. A.I_usage_tracker §4.8 아키텍처를 그대로 차용한다.

```
[개발자] git tag v*.*.* → git push --tags
    ↓
[GitHub Actions release.yml 자동 트리거]
    ↓
dotnet publish (self-contained win-x64 single-file)
    ↓
Inno Setup 컴파일 → AgentPaw_Setup_v*.exe
    ↓
Get-FileHash SHA256 → AgentPaw_Setup_v*.exe.sha256
    ↓
GitHub Release 생성 + 인스톨러·sidecar 업로드
    ↓
[사용자 PC UpdateService 24h 폴링 감지]
    ↓
다운로드 → SHA256 검증 → 무인 설치 (/SILENT)
```

## 2. 필수 파일 구성

. `AgentPaw/AgentPaw.csproj` : 버전 속성 4종 정의
. `installer/AgentPaw.iss` : Inno Setup 스크립트
. `.github/workflows/release.yml` : 릴리스 자동화 파이프라인
. `AgentPaw/Services/UpdateService.cs` : 업데이트 폴링·검증·실행 로직

## 3. 버전 동기화 규칙

버전을 변경할 때 아래 위치를 **모두 동기화**한다. 하나라도 누락되면 런타임 `CurrentVersion`·인스톨러·Release 태그 간 불일치가 발생한다.

| 파일 | 위치 | 형식 | 갱신 주체 |
|---|---|---|---|
| `AgentPaw/AgentPaw.csproj` | `<Version>` | `X.Y.Z` | 개발자 수동 |
| `AgentPaw/AgentPaw.csproj` | `<AssemblyVersion>` | `X.Y.Z.0` | 개발자 수동 |
| `AgentPaw/AgentPaw.csproj` | `<FileVersion>` | `X.Y.Z.0` | 개발자 수동 |
| `AgentPaw/AgentPaw.csproj` | `<InformationalVersion>` | `X.Y.Z` | 개발자 수동 |
| `installer/AgentPaw.iss` | `AppVersion` | `X.Y.Z` | release.yml 자동 치환 |
| `installer/AgentPaw.iss` | `AppVerName` | `AgentPaw X.Y.Z` | release.yml 자동 치환 |
| `installer/AgentPaw.iss` | `OutputBaseFilename` | `AgentPaw_Setup_vX.Y.Z` | release.yml 자동 치환 |
| Git 태그 | `vX.Y.Z` | `vX.Y.Z` | 개발자 수동 |

단, `.csproj` 만 수동 관리하고 `.iss` 는 GitHub Actions 에서 태그 값으로 자동 덮어쓴다.

## 4. SemVer 적용 원칙

CLAUDE.md §6.1 의 3단계 버전 체계를 앱 버전에도 동일 적용한다.

. **Major** : UI 전체 재편, 데이터 스키마 호환성 단절, 오케스트레이션 모델 변경
. **Minor** : 신규 Page·Service 추가, Chat Bot 플랫폼 추가, 기능 모듈 신설
. **Patch** : 버그 수정, 문구 변경, 내부 리팩터링

단, v1.0.0 미만 프리릴리스 구간에서는 Minor 도 Breaking Change 를 포함할 수 있다.

## 5. 배포 단계별 체크리스트

### 5.1. 릴리스 전 검증

. 로컬에서 `dotnet publish -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true` 성공 확인
. 생성된 `AgentPaw.exe` 를 실제 실행하여 스플래시·Settings·DI 초기화 정상 동작 확인
. `UpdateService.CheckForUpdateAsync()` 가 현재 버전에서 null 반환함을 확인 (무한 업데이트 루프 방지)
. `appsettings.Development.json`·`*.pfx`·`secrets.json` 가 publish 결과물에 포함되지 않음을 확인 (`.gitignore` 검증)

### 5.2. 태그 푸시

```bash
git tag v0.5.0
git push origin v0.5.0
```

### 5.3. GitHub Actions 모니터링

. Actions 탭에서 `Release` 워크플로 진행 상황 확인
. 실패 시 로그 검토 → 원인 분류 (publish 실패 / Inno Setup 실패 / 업로드 실패)
. 성공 시 `https://github.com/zitify-blip/Agent_Paw/releases/tag/v*.*.*` 접근 확인

### 5.4. 사후 검증

. Release 페이지에서 `AgentPaw_Setup_v*.exe` 와 `.sha256` 파일 존재 확인
. 수동으로 인스톨러 다운로드 → 설치 → 앱 실행 → Settings 에서 버전 표시 일치 확인
. 이전 버전 앱에서 "업데이트 확인" 클릭 → 신버전 감지·다운로드·검증·재설치 전체 흐름 검증

## 6. 보안·무결성

. 인스톨러 파일은 **코드 서명 없음** — SmartScreen 경고 발생. "추가 정보 → 실행" 우회 필요
. 게시자 신원 보장 수단은 **SHA256 sidecar 검증** 하나뿐 → 다운로드 무결성만 담보
. 사용자에게 신원 보장이 필요해지는 시점에 Azure Trusted Signing 또는 OV/EV Code Signing 으로 업그레이드
. Release 자산은 HTTPS + GitHub 인프라에 저장되므로 중간자 공격 위험은 낮다
. UpdateService 는 다운로드 URL 이 `https://github.com/` 로 시작하지 않으면 거부한다 (URL injection 방지)

## 7. 긴급 롤백 절차

배포 후 심각한 결함이 발견된 경우 아래 순서로 대응한다.

1. GitHub Releases 페이지에서 문제 릴리스를 **Pre-release 로 전환** (자동 업데이트 대상에서 제외)
2. `UpdateService` 는 `tag_name` 이 "latest" 인 릴리스만 조회하므로, Pre-release 전환으로 즉시 노출 차단
3. 직전 안정 버전을 재태그하거나, 수정된 Patch 버전을 긴급 릴리스
4. 긴급 릴리스 후 기존 Pre-release 는 삭제하지 않고 보존 (감사·회귀 추적 목적, CLAUDE.md §6.4 준용)

단, `latest` 태그 자동 선정 로직은 GitHub API 의 `releases/latest` 엔드포인트가 최신 비-Pre-release 릴리스를 반환한다는 규약에 의존한다.

## 8. 향후 과제

. **Delta 업데이트** : 현재는 전체 인스톨러 재다운로드 방식. 대용량화 시 `bsdiff` 기반 delta patch 검토
. **다중 채널** : stable / beta / canary 채널 분리 — Release tag 에 `-beta`·`-canary` suffix 로 구분
. **롤백 UI** : 사용자가 "이전 버전으로 되돌리기" 버튼으로 안정 버전 재설치 가능하게
. **업데이트 알림 실시간화** : 24h 폴링 대신 GitHub Webhook + Azure Function 으로 실시간 푸시
. **코드 서명 도입** : 배포 규모 10명 이상 시 Azure Trusted Signing 구독 전환 검토

---

## 1. 표준 루트 구조

```
project-root/
├── src/              ← 애플리케이션 소스 코드
├── tests/            ← 테스트 코드
├── docs/             ← 프로젝트 문서
├── scripts/          ← 빌드·배포·유틸리티 스크립트
├── config/           ← 환경별 설정 파일
├── public/           ← 정적 자원 (웹 프로젝트 한정)
├── .env.example      ← 환경변수 예시 파일
├── .gitignore
├── README.md
└── package.json      ← 또는 pyproject.toml / pom.xml 등
```

## 2. 각 디렉토리 책임

| 디렉토리 | 책임 | 포함 금지 대상 |
|---|---|---|
| `src/` | 프로덕션 애플리케이션 코드 | 테스트 코드, 임시 파일 |
| `tests/` | 단위·통합·E2E 테스트 | 프로덕션 로직 |
| `docs/` | 산출물 문서(MD/DOCX) | 소스 코드, 설정 파일 |
| `scripts/` | 빌드·배포·마이그레이션 스크립트 | 애플리케이션 핵심 로직 |
| `config/` | 환경별 설정(JSON/YAML) | 시크릿, API 키 |
| `public/` | 정적 자원(이미지·폰트·favicon) | 소스 코드 |

## 3. src 내부 구조

### 3.1. 레이어드 구조 (백엔드)

```
src/
├── controllers/   ← 요청·응답 처리
├── services/      ← 비즈니스 로직
├── repositories/  ← 데이터 접근
├── models/        ← 도메인 모델·엔티티
├── middlewares/   ← 미들웨어·필터
├── utils/         ← 공용 유틸리티
└── types/         ← 타입 정의 (TS 한정)
```

### 3.2. 기능별 구조 (프론트엔드)

```
src/
├── components/    ← 재사용 UI 컴포넌트
├── pages/         ← 페이지 단위 컴포넌트
├── hooks/         ← 커스텀 훅 (React)
├── store/         ← 전역 상태 관리
├── api/           ← API 클라이언트
├── styles/        ← 전역 스타일
└── assets/        ← 번들에 포함되는 자원
```

## 4. tests 구조

. `tests/` 하위 구조는 `src/` 구조를 미러링한다.
. 테스트 파일은 대상 소스 파일과 동일 이름에 `.test` / `.spec` 접미사를 붙인다.

```
tests/
├── unit/          ← 단위 테스트
├── integration/   ← 통합 테스트
└── e2e/           ← End-to-End 테스트
```

단, 언어 관례(Python의 `tests_*.py` 등)가 있는 경우 해당 관례를 따른다.

## 5. 배치 원칙

. 단일 책임 : 각 디렉토리는 하나의 관심사만 담당한다.
. 순환 참조 금지 : 상위 레이어가 하위 레이어를 참조한다. 역방향 참조는 불가하다.
. 깊이 제한 : 디렉토리 깊이는 최대 4단계로 제한한다. 초과 시 모듈 분리를 검토한다.
. 공용 모듈 격리 : 여러 모듈에서 사용하는 코드는 `common/` 또는 `shared/` 에 배치한다.

## 6. 금지 배치

. 소스 코드 루트(`src/`)에 설정 파일·문서 파일 배치를 금지한다.
. 테스트 파일을 `src/` 에 함께 두는 것을 금지한다.
단, 컴포넌트 단위 병행 배치(`Component.tsx` + `Component.test.tsx`) 는 팀 합의 시 허용한다.

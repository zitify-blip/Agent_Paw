---

## 1. 로그 레벨

| 레벨 | 용도 | 예시 | 환경별 출력 |
|---|---|---|---|
| `DEBUG` | 개발·디버깅 상세 정보 | 변수 값·함수 진입 추적 | dev 전용 |
| `INFO` | 정상 동작·이벤트 기록 | 로그인 성공·주문 생성 | dev/stg/prd |
| `WARN` | 주의 필요·잠재 문제 | 재시도 발생·deprecated API 호출 | dev/stg/prd |
| `ERROR` | 처리 실패·예외 발생 | DB 연결 실패·비즈니스 예외 | dev/stg/prd |
| `FATAL` | 시스템 중단 수준 치명 오류 | OOM·프로세스 종료 | dev/stg/prd + 알림 |

. 프로덕션(prd) 기본 레벨은 `INFO` 이다. 긴급 진단 시 `DEBUG` 로 일시 상향한다.
. `DEBUG` 로그는 상용 환경에 상시 출력하는 것을 금지한다.

## 2. 로그 포맷

. 구조화 로그(JSON) 를 기본으로 사용한다. 텍스트 로그는 개발 환경에 한정한다.
. 필수 필드 : `timestamp`, `level`, `logger`, `message`, `trace_id`, `user_id`.
. 타임스탬프는 ISO 8601 UTC 형식을 사용한다. 예) `2026-04-16T09:30:15.123Z`.

```json
{
  "timestamp": "2026-04-16T09:30:15.123Z",
  "level": "INFO",
  "logger": "OrderService",
  "message": "주문 생성 성공",
  "trace_id": "abc-123-def",
  "user_id": "U20260416001",
  "order_id": "O20260416001"
}
```

## 3. 메시지 작성 규칙

. 메시지는 "주어 + 동작 + 결과" 구조로 작성한다.
. 동적 값은 메시지 본문이 아닌 별도 필드로 분리한다.
. 에러 로그는 반드시 stack trace를 포함한다.

```
❌  System.out.println("user: " + userId + " logged in")
✅  logger.info("User 로그인 성공", { user_id: userId, ip: clientIp })
```

## 4. 민감정보 마스킹

| 항목 | 마스킹 방식 | 예시 |
|---|---|---|
| 비밀번호 | 전체 마스킹 | `********` |
| 주민등록번호 | 뒷자리 마스킹 | `900101-1******` |
| 신용카드번호 | 중간 마스킹 | `4123-****-****-1234` |
| 전화번호 | 중간 마스킹 | `010-****-5678` |
| 이메일 | 앞자리 일부만 노출 | `us**@example.com` |
| API 키·토큰 | 접두 4자만 노출 | `sk_li_xxxx...` |
| 주소 | 동 단위까지만 기록 | `서울시 강남구 역삼동` |

. 마스킹은 로깅 시점에 System이 자동 처리한다. 수동 마스킹을 금지한다.
. 마스킹 누락은 보안 사고로 간주한다.

## 5. 로그 작성 위치

. 요청 진입·응답 : Controller 계층에서 `INFO` 로 기록한다.
. 비즈니스 이벤트 : Service 계층에서 `INFO` 로 기록한다.
. 외부 연동 : API 호출·응답을 `INFO` 로 기록한다. 실패 시 `ERROR` 로 기록한다.
. 예외 발생 : catch 블록에서 `ERROR` 로 stack trace와 함께 기록한다.

## 6. 금지 사항

. `console.log`, `print`, `System.out.println` 직접 사용을 금지한다. 지정 로거를 사용한다.
. 민감정보(비밀번호·카드번호·토큰)의 원문 로깅을 절대 금지한다.
. 반복문 내부에서 대량 로그를 출력하는 것을 금지한다. 샘플링한다.
. 로그 레벨을 조건 분기로 제어하는 것을 금지한다. 로거 설정으로 제어한다.

단, 금융·의료 등 규제 산업에서는 관련 법규에 따른 추가 마스킹·보관 기준을 적용한다.

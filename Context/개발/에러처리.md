---

## 1. 예외 클래스 계층

```
AppException (최상위)
├── BusinessException       ← 비즈니스 규칙 위반
│   ├── ValidationException ← 입력 검증 실패
│   ├── NotFoundException   ← 자원 없음
│   └── ConflictException   ← 상태 충돌
├── SystemException         ← 시스템 자체 오류
│   ├── DatabaseException
│   ├── ExternalApiException
│   └── ConfigurationException
└── SecurityException       ← 인증·인가 오류
    ├── UnauthorizedException ← 인증 실패
    └── ForbiddenException    ← 권한 부족
```

. 모든 커스텀 예외는 `AppException` 을 상속한다.
. 언어별 기본 예외(`Exception`, `Error`) 를 직접 throw 하는 것을 금지한다.

## 2. 에러 코드 체계

```
[E/W][카테고리 2자리][세부 3자리]
```

| 접두 | 카테고리 | 예시 코드 | 의미 |
|---|---|---|---|
| `E` | Error | `E10001` | 치명 오류 |
| `W` | Warning | `W20001` | 경고성 오류 |
| `10` | 입력 검증 | `E10001` | 필수값 누락 |
| `20` | 인증·인가 | `E20001` | 인증 실패 |
| `30` | 자원 상태 | `E30001` | 자원 없음 |
| `40` | 외부 연동 | `E40001` | 외부 API 타임아웃 |
| `50` | DB | `E50001` | DB 연결 실패 |
| `90` | 시스템 | `E90001` | 설정 오류 |

. 에러 코드는 중앙 레지스트리(`error-codes.yaml`) 에서 유일성을 관리한다.
. 코드 신설 시 PR 리뷰에서 중복 여부를 확인한다.

## 3. try-catch 원칙

### 3.1. catch 대상

. 예상 가능한 예외만 catch한다. 포괄 catch(`catch (e)`, `except:`)를 금지한다.
. 구체 예외 타입을 우선 catch한다. 상위 예외는 마지막에 배치한다.

```
❌  try { doSomething() } catch (e) { /* 모든 예외 삼킴 */ }
✅  try { doSomething() }
    catch (NotFoundException e) { /* 구체 처리 */ }
    catch (DatabaseException e) { /* 시스템 오류 별도 처리 */ }
```

### 3.2. 재throw·변환

. 하위 계층 예외를 상위 계층 예외로 래핑하여 재throw 한다.
. 원인 예외(cause)를 반드시 포함한다. 스택 트레이스를 유실하지 않는다.

```
throw new OrderCreationException("주문 생성 실패", originalException);
```

### 3.3. 금지 패턴

. 빈 catch 블록(예외 삼킴)을 절대 금지한다.
. catch에서 로깅만 하고 throw하지 않는 것을 금지한다. 상위가 인지하지 못한다.
. 정상 흐름 제어에 예외를 사용하는 것을 금지한다.

단, 재시도 로직·폴백 처리에서는 catch 후 정상 진행이 가능하다. 이 경우 반드시 `WARN` 로그를 남긴다.

## 4. 에러 응답 포맷

API 응답 에러는 표준 포맷을 따른다.

```json
{
  "success": false,
  "error": {
    "code": "E10001",
    "message": "필수 파라미터가 누락되었습니다",
    "field": "email",
    "trace_id": "abc-123-def"
  }
}
```

| 필드 | 필수 | 설명 |
|---|---|---|
| `code` | Y | 에러 코드 |
| `message` | Y | 사용자 노출 메시지 |
| `field` | N | 검증 실패 필드 (Validation 에러 한정) |
| `trace_id` | Y | 로그 추적 ID |
| `detail` | N | 개발 환경 전용 상세 정보 |

단, 프로덕션 환경에서는 내부 구조(스택 트레이스·쿼리) 를 응답에 포함하는 것을 절대 금지한다.

## 5. HTTP 상태 코드 매핑

| 예외 유형 | HTTP Status |
|---|---|
| ValidationException | 400 Bad Request |
| UnauthorizedException | 401 Unauthorized |
| ForbiddenException | 403 Forbidden |
| NotFoundException | 404 Not Found |
| ConflictException | 409 Conflict |
| BusinessException (기타) | 422 Unprocessable Entity |
| ExternalApiException | 502 Bad Gateway |
| SystemException | 500 Internal Server Error |

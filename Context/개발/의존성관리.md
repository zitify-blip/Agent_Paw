---

## 1. 패키지 추가 승인 절차

1. 작성자는 신규 패키지 추가 필요성을 이슈로 등록한다.
2. 아래 승인 기준표에 따라 검토한다.
3. 테크리드 또는 아키텍트가 승인한다.
4. 승인 후 PR을 통해 추가한다.

### 1.1. 승인 기준

| 기준 | 요구사항 |
|---|---|
| 라이선스 | MIT / Apache 2.0 / BSD 등 상용 호환 라이선스 |
| 활성도 | 최근 6개월 이내 커밋 활동 존재 |
| 다운로드 수 | 주간 10,000건 이상 또는 대안 부재 |
| 유지보수 | 최근 1년 내 이슈 대응 이력 존재 |
| 취약점 | CVE 기준 Critical·High 이슈 없음 |
| 대안 검토 | 내부 구현·표준 라이브러리로 대체 가능성 검토 완료 |

. GPL·AGPL·SSPL 라이선스 패키지는 원칙적으로 금지한다.
단, 법무팀 검토 후 사내 비공개 환경 한정으로 예외 승인이 가능하다.

## 2. 버전 고정 규칙

| 생태계 | Lock 파일 | 커밋 포함 |
|---|---|---|
| npm / yarn / pnpm | `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml` | ✅ 필수 |
| Python | `requirements.txt` (pinned) 또는 `poetry.lock` | ✅ 필수 |
| Java / Kotlin | `gradle.lockfile` 또는 Maven BOM | ✅ 필수 |
| Go | `go.sum` | ✅ 필수 |
| Ruby | `Gemfile.lock` | ✅ 필수 |

. 의존성 버전은 정확 버전(exact version) 을 명시한다. `^`, `~` 등 범위 지정을 지양한다.
. Lock 파일을 `.gitignore` 에 포함하는 것을 금지한다.
. Lock 파일 변경은 PR에 의존성 변경 사유를 명시한다.

## 3. 취약점 점검

### 3.1. 자동 점검

. CI 파이프라인에 취약점 스캐너를 필수 포함한다.
. 점검 도구 : `npm audit`, `pip-audit`, `OWASP Dependency Check`, `Snyk`, `Dependabot`.
. Critical·High 취약점 발견 시 PR을 차단한다.

### 3.2. 주기 점검

. 주간 : System이 의존성 스캔 리포트를 발행한다.
. 월간 : 보안 담당자가 리포트를 검토하고 업그레이드 계획을 수립한다.
. 분기 : 사용 중인 전체 의존성 목록을 재검토한다.

### 3.3. 대응 우선순위

| 취약점 등급 | 대응 기한 | 조치 |
|---|---|---|
| Critical | 24시간 이내 | 긴급 패치 배포 |
| High | 7일 이내 | 계획 패치 |
| Medium | 30일 이내 | 차기 릴리즈 반영 |
| Low | 90일 이내 | 정기 업그레이드 반영 |

## 4. 업그레이드 원칙

. Patch 버전 업그레이드는 Dependabot 자동 PR을 허용한다.
. Minor 버전 업그레이드는 테스트 통과 확인 후 병합한다.
. Major 버전 업그레이드는 Breaking Change 검토 후 별도 이슈로 진행한다.

## 5. 금지 사항

. 검증되지 않은 개인 포크·사내 미등록 저장소에서 패키지를 설치하는 것을 금지한다.
. 프로덕션 의존성(`dependencies`) 과 개발 의존성(`devDependencies`) 을 혼동 등록하는 것을 금지한다.
. 전역 설치(`npm install -g`) 의존을 프로젝트에 포함하는 것을 금지한다.
. 의존성 설치 스크립트(`postinstall` 등) 내부에서 네트워크 요청·파일 쓰기를 수행하는 패키지 사용을 엄격히 제한한다.

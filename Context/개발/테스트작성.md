---

## 1. 테스트 종류

| 종류 | 범위 | 도구 예시 | 실행 속도 | 작성 의무 |
|---|---|---|---|---|
| 단위(Unit) | 함수·클래스 단위 | Jest, pytest, JUnit | 밀리초 | ✅ 필수 |
| 통합(Integration) | 모듈 간 연동 | Supertest, TestContainers | 수 초 | ✅ 필수 |
| E2E | 전체 사용자 시나리오 | Playwright, Cypress | 수십 초 | ⚠️ 주요 시나리오 한정 |
| 성능(Performance) | 부하·응답시간 | k6, JMeter | 수 분 | ⚠️ 릴리즈 전 |

## 2. 커버리지 기준

| 대상 | 라인 커버리지 | 브랜치 커버리지 | 비고 |
|---|---|---|---|
| 비즈니스 로직(Service) | 80% 이상 | 70% 이상 | CI 게이트 |
| 도메인 모델 | 90% 이상 | 80% 이상 | 순수 함수 위주 |
| 유틸리티 | 90% 이상 | 80% 이상 | |
| Controller | 60% 이상 | - | 통합 테스트로 보완 |
| 전체 평균 | 75% 이상 | 65% 이상 | |

. 커버리지 미달 PR은 CI에서 차단한다.
. 외부 라이브러리 래퍼·자동 생성 코드는 커버리지 산정에서 제외한다.

단, 프로토타입·실험 코드는 커버리지 기준을 면제한다. 단, `@experimental` 태그를 명시한다.

## 3. 테스트명 규칙

. 형식 : `[대상]_[조건]_[기대결과]` 또는 `should [기대결과] when [조건]`.
. 한국어·영어 작성을 모두 허용한다. 프로젝트 내 통일한다.
. 테스트명은 테스트 실패 시 원인을 유추할 수 있도록 구체적으로 작성한다.

```
❌  test_login()
❌  test1()
✅  test_login_whenEmailIsEmpty_thenThrowValidationException
✅  should_ThrowValidationException_when_EmailIsEmpty
✅  로그인_이메일이_비어있으면_ValidationException을_던진다
```

## 4. 테스트 구조 (AAA 패턴)

. Arrange : 테스트 데이터·Mock 준비
. Act : 테스트 대상 실행
. Assert : 결과 검증

```javascript
it('주문 생성 시 재고가 부족하면 OutOfStockException을 던진다', () => {
  // Arrange
  const product = { id: 'P001', stock: 0 };
  const service = new OrderService(mockRepo);

  // Act & Assert
  expect(() => service.createOrder(product, 1))
    .toThrow(OutOfStockException);
});
```

## 5. 테스트 원칙 (F.I.R.S.T)

| 원칙 | 의미 |
|---|---|
| Fast | 단위 테스트는 100ms 이내에 종료한다 |
| Independent | 테스트 간 실행 순서 의존을 금지한다 |
| Repeatable | 어떤 환경에서도 동일 결과를 보장한다 |
| Self-validating | pass/fail 로 자동 판정한다. 육안 확인을 금지한다 |
| Timely | 프로덕션 코드와 동시에 작성한다 |

## 6. Mock·Stub 사용

. 외부 의존(DB·API·파일시스템)은 단위 테스트에서 반드시 Mock 처리한다.
. 통합 테스트는 실제 의존(또는 TestContainer)을 사용한다.
. Mock 은 동작 검증(behavior) 용, Stub 은 상태 검증(state) 용으로 구분한다.

## 7. 금지 사항

. 테스트 간 전역 상태 공유를 금지한다. beforeEach·setUp 에서 초기화한다.
. 실제 프로덕션 DB·외부 API를 테스트에서 호출하는 것을 절대 금지한다.
. 테스트 결과가 비결정적(시간·랜덤 의존)인 테스트를 금지한다. 고정값을 주입한다.
. `skip`, `only`, `@Ignore` 를 커밋하는 것을 금지한다.

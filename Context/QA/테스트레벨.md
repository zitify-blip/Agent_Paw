---

## 1. 테스트 레벨 개요

4개 레벨은 아래 방향으로 점진적으로 확장된다. 상위 레벨은 하위 레벨 통과를 전제한다.

| 레벨 | 검증 대상 | 주체 | 실행 시점 | 실행 환경 |
|---|---|---|---|---|
| Unit | 함수·클래스 단일 단위 | 개발자 | 코드 커밋 시 | 로컬 + CI |
| Integration | 모듈 간 연동, DB·API 경계 | 개발자 + QA | PR 머지 전 | DEV |
| System(E2E) | 전체 사용자 여정, UI부터 DB까지 | QA | 일 1회, 릴리즈 전 | QA |
| UAT | 비즈니스 요구사항 충족 여부 | PM + 현업 | 릴리즈 D-3 ~ D-1 | STG |

---

## 2. 레벨별 상세

### 2.1. Unit Test

. 검증 범위 : 개별 함수·메서드의 입출력, 예외 처리, 경계값을 검증한다.
. 책임 주체 : 개발자가 프로덕션 코드와 동시에 작성한다.
. 커버리지 기준 : 신규 코드 라인 커버리지 80% 이상, 브랜치 커버리지 70% 이상을 유지한다.
. 실행 시점 : 로컬 커밋 시 자동 실행되며, CI 파이프라인에서 재실행된다.
. 실행 시간 기준 : 전체 Unit 스위트는 5분 이내에 완료되어야 한다.

단, 외부 의존성(DB·API·파일)은 Mock·Stub으로 격리한다. 실제 연동은 Integration 레벨에서 검증한다.

### 2.2. Integration Test

. 검증 범위 : 2개 이상 모듈·서비스 간 호출, DB CRUD 흐름, 외부 API 연동을 검증한다.
. 책임 주체 : 개발자가 주도하고 QA가 시나리오 검토·보강한다.
. 실행 시점 : PR 머지 전 CI 파이프라인에서 자동 실행되며, 실패 시 머지가 차단된다.
. 데이터 처리 : System이 테스트 시작 시 고정 시드 데이터를 INSERT하고, 종료 시 트랜잭션을 롤백한다.

### 2.3. System Test (E2E)

. 검증 범위 : 사용자 여정 전체(Client → Server → Database → 외부 연동)를 종단 검증한다.
. 책임 주체 : QA가 TC 설계·실행·결과 분석을 주관한다.
. 실행 시점 : 야간 빌드(매일 01:00) 자동 실행, 릴리즈 후보 빌드 생성 시 수동 실행한다.
. 환경 : QA 환경에서 실행하며, 실제 프로덕션 데이터 구조와 동일한 익명화 데이터셋을 사용한다.

단, 외부 결제·SMS 게이트웨이는 Sandbox 계정을 사용하여 실제 과금·발송을 방지한다.

### 2.4. User Acceptance Test (UAT)

. 검증 범위 : 비즈니스 요구사항(기획서·사용자 스토리) 충족 여부를 현업 관점에서 검증한다.
. 책임 주체 : PM이 주관하고 현업 대표 사용자가 실행한다. QA는 기술 지원을 담당한다.
. 실행 시점 : 릴리즈 D-3 ~ D-1, 3일간 진행한다.
. 환경 : STG 환경. 실제 운영 데이터 이관본을 사용한다.
. 종료 조건 : P0·P1 결함 0건 + 현업 사인오프 완료 시 종료된다.

---

## 3. 레벨 간 이관 기준

하위 레벨 완료 없이 상위 레벨로 진입할 수 없다. System이 CI에서 이관 기준을 검증한다.

| 이관 방향 | 필수 조건 |
|---|---|
| Unit → Integration | Unit 스위트 100% 통과 + 커버리지 기준 충족 |
| Integration → System | Integration 스위트 100% 통과 + PR 리뷰 승인 |
| System → UAT | E2E 리그레션 100% 통과 + P0 결함 0건 |
| UAT → 릴리즈 | 현업 사인오프 + Go/No-Go 회의 승인 |

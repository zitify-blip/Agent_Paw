## 1. 전략 기본 원칙

### 1.1. Shift-Left 원칙

QA는 배포 직전 검증이 아닌 요구사항 정의 시점부터 개입한다. 결함 조기 발견이 전체 비용을 낮춘다.

| 단계 | QA 개입 활동 | 산출물 |
|---|---|---|
| 요구사항 | QA가 기획서 리뷰에 참여하여 테스트 가능성(Testability)을 검증한다 | 리뷰 코멘트, 명확화 요청 |
| 설계 | QA가 FSD·API 명세 리뷰에 참여하여 경계값·예외 흐름을 식별한다 | 테스트 시나리오 초안 |
| 개발 | 개발자가 Unit Test를 작성하고, QA가 Integration TC를 병행 작성한다 | Unit Test 코드, TC 문서 |
| 빌드 | System이 CI 파이프라인에서 자동 리그레션을 실행한다 | 자동화 테스트 리포트 |
| 배포 | QA가 UAT·릴리즈 검증을 주관한다 | 릴리즈 검증 리포트 |

### 1.2. 테스트 피라미드

테스트 자산은 하단이 넓고 상단이 좁은 피라미드 구조를 유지한다. 단위 테스트 비중이 높을수록 피드백 주기가 짧다.

| 레벨 | 비중(권장) | 주체 | 실행 주기 |
|---|---|---|---|
| Unit | 70% | 개발자 | 커밋마다 |
| Integration | 20% | 개발자 + QA | PR 머지 시 |
| System(E2E) | 8% | QA | 일 1회(야간) |
| UAT | 2% | PM + 현업 | 릴리즈 전 |

단, 레거시 모듈로 Unit 커버리지 확보가 불가한 영역은 Integration·E2E 비중을 한시적으로 상향 조정한다.

### 1.3. 리스크 기반 테스트

모든 기능을 동일 강도로 검증하는 것은 비효율적이다. 리스크(발생 가능성 × 영향도)에 따라 테스트 깊이를 차등 적용한다.

| 리스크 등급 | 조건 | 테스트 범위 |
|---|---|---|
| High | 결제·정산·인증 등 금전·보안 직결 기능 | 전체 TC 100% + 네거티브 + 성능 |
| Medium | 핵심 사용자 여정(주요 CRUD, 대시보드) | 기능 TC 100% + 경계값 |
| Low | 부가 기능(통계, 내보내기, UI 편의) | Happy Path TC만 실행 |

## 2. 전략 실행 체계

### 2.1. QA 활동 RACI

| 활동 | 개발자 | QA | PM | 현업 |
|---|---|---|---|---|
| Unit Test 작성 | R | C | I | - |
| TC 설계·문서화 | C | R | A | C |
| 자동화 스크립트 작성 | C | R | I | - |
| 리그레션 실행 | - | R | A | - |
| UAT 주관 | I | C | R | R |
| 릴리즈 Go/No-Go 판단 | C | C | R | A |

※ R=실행 A=승인 C=협의 I=통보

### 2.2. 전략 갱신 주기

. 분기 단위 리뷰 : QA가 분기 말 결함 통계·리그레션 효과를 분석하여 전략을 재조정한다.
. 주요 장애 발생 후 : Sev1 장애 발생 시 원인 분석(RCA) 결과를 토대로 테스트 전략을 즉시 보강한다.

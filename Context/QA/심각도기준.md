결함의 심각도를 Sev1~Sev4 4단계로 분류한다. 심각도는 "결함이 시스템·사용자에 미치는 영향의 크기"를 나타내며, 처리 시한을 의미하는 우선순위(P0~P3)와 독립적으로 판단한다.

---

## 1. 심각도 분류 기준

### 1.1. 전체 매트릭스

| 심각도 | 정의 | 영향 범위 | 데이터 무결성 |
|---|---|---|---|
| Sev1(Critical) | 서비스 전체 또는 핵심 기능이 완전히 동작 불가 | 전체 사용자 | 데이터 손실·오염 발생 가능 |
| Sev2(Major) | 주요 기능이 부분 동작 불가. 우회 경로 존재 | 50% 이상 사용자 | 데이터는 정상 |
| Sev3(Minor) | 부가 기능 오류. 핵심 흐름에 영향 없음 | 일부 사용자 | 데이터는 정상 |
| Sev4(Trivial) | UI·오탈자·편의성 이슈 | 단일 사용자 또는 시각적 | 데이터는 정상 |

### 1.2. 우선순위와의 관계

심각도와 우선순위는 원칙적으로 독립이지만, 일반적인 매핑 관계는 아래와 같다.

| 심각도 | 일반 우선순위 | 예외 케이스 |
|---|---|---|
| Sev1 | P0 | 영향 사용자가 극소수면 P1로 하향 가능 |
| Sev2 | P1 | 핵심 사용자 여정이면 P0로 상향 |
| Sev3 | P2 | 다수 누적 시 P1로 상향 |
| Sev4 | P3 | VIP 고객 보고 건은 P2로 상향 |

## 2. 심각도별 예시

| 심각도 | 예시 상황 |
|---|---|
| Sev1 | 로그인 API 전면 실패, 결제 금액이 실제와 다르게 기록됨, DB 커넥션 풀 고갈 |
| Sev2 | 파일 업로드 10MB 이상 실패, 특정 권한 사용자 페이지 접근 불가, 대시보드 일부 지표 미노출 |
| Sev3 | 정렬 순서 불일치, 툴팁 미노출, 검색 결과 페이징 오류 |
| Sev4 | 버튼 라벨 오탈자, 아이콘 정렬 1px 어긋남, 하이라이트 색상 차이 |

## 3. 에스컬레이션 체계

### 3.1. 자동 에스컬레이션

System이 결함 심각도 등록 시점에 아래 규칙으로 자동 에스컬레이션한다.

| 심각도 | 통보 대상 | 통보 채널 | 통보 시점 |
|---|---|---|---|
| Sev1 | 개발팀 전체 + PM + QA 리드 + CTO | Slack DM + SMS | 즉시 |
| Sev2 | 개발팀 리드 + PM + QA 리드 | Slack 채널 | 10분 이내 |
| Sev3 | 담당 개발자 + QA 담당 | Slack 채널 | 30분 이내 |
| Sev4 | 담당 개발자 | JIRA 알림 | 1시간 이내 |

### 3.2. 수동 에스컬레이션 기준

. Sev3 결함이 48시간 내 3건 이상 누적 : QA가 Sev2로 상향 조정한다.
. 동일 모듈에서 Sev2 결함 연속 발생 : QA 리드가 해당 모듈 전체 리뷰를 요청한다.
. Sev1 발생 후 RCA 미완료 : 72시간 경과 시 CTO에게 직접 보고한다.

---

## 4. 심각도 판정 절차

. 1차 판정 : 결함 등록자(QA 또는 개발자)가 본 기준에 따라 초기 심각도를 지정한다.
. 2차 검토 : QA 리드가 일 1회 신규 등록 결함을 리뷰하여 심각도를 확정한다.
. 재조정 : 수정 중 영향 범위가 확장된 경우 담당 개발자가 QA 리드에게 재조정을 요청한다.

단, Sev1 판정은 반드시 QA 리드 + PM 2인 이상의 동의를 거친다.

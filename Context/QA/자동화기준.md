테스트 자동화 대상 선정 기준, 도구 스택, 유지보수 책임을 정의한다. 자동화는 수동 테스트를 완전 대체하는 것이 아닌, 반복·회귀 영역을 우선적으로 대체하는 것을 원칙으로 한다.

---

## 1. 자동화 선정 원칙

### 1.1. 3대 기준

| 기준 | 정의 | 판정 방법 |
|---|---|---|
| 반복성 | 동일 TC가 릴리즈마다 반복 실행되는가 | 월 2회 이상 실행 시 적합 |
| 안정성 | TC 결과가 환경 요인에 영향받지 않는가 | Flaky 비율 5% 이하 시 적합 |
| ROI | 자동화 투입 대비 절감 시간이 큰가 | 3개월 내 손익분기 시 적합 |

3대 기준 중 2개 이상 충족 시 자동화 후보로 등록한다.

### 1.2. 자동화 적합·부적합 영역

| 영역 | 자동화 적합도 | 사유 |
|---|---|---|
| API 기능 테스트 | 적합 | 입출력 명확, 실행 시간 짧음 |
| 핵심 사용자 여정 E2E | 적합 | 반복 실행 빈도 높음 |
| 리그레션 스위트 | 적합 | 매 빌드 반복 실행 |
| 성능·부하 테스트 | 적합 | 수동 실행이 현실적으로 불가 |
| 신규 UI 탐색 테스트 | 부적합 | 변경 빈도 높아 유지보수 비용 초과 |
| 사용성(UX) 평가 | 부적합 | 주관적 판단 필요 |
| 일회성 이벤트·마이그레이션 | 부적합 | 반복성 없음 |

## 2. 자동화 도구 스택

### 2.1. 표준 도구

| 계층 | 도구 | 용도 | 담당 |
|---|---|---|---|
| Unit | Jest / JUnit | 함수 단위 검증 | 개발자 |
| API | Postman + Newman | REST API 검증 | QA |
| E2E 웹 | Playwright | 웹 UI 여정 검증 | QA |
| E2E 모바일 | Appium | iOS·Android UI 검증 | QA |
| 성능 | k6 | 부하·스트레스 | QA + SRE |
| CI 통합 | GitHub Actions | 자동 실행 오케스트레이션 | DevOps |

### 2.2. 도구 선정 절차

. 신규 도구 도입 시 : QA가 POC를 2주간 진행하고, QA 리드·DevOps 승인 후 도입한다.
. 도구 교체 시 : 기존 스크립트 이관 비용과 신규 도구 학습 비용을 동시 산정하여 ROI 6개월 이내일 때만 교체한다.
. 중복 도구 사용 금지 : 동일 계층에서 2개 이상의 도구 병행 사용을 원칙적으로 불가하다.

---

## 3. 자동화 스크립트 품질 기준

### 3.1. 필수 품질 요건

| 요건 | 기준 | 검증 방법 |
|---|---|---|
| 단일 책임 | 1개 TC = 1개 검증 목적 | 코드 리뷰 |
| 독립성 | TC 간 실행 순서 의존성 없음 | 무작위 순서 실행 통과 |
| 멱등성 | 반복 실행 시 동일 결과 보장 | 연속 3회 실행 통과 |
| 실행 시간 | 단일 API TC 5초 이내, E2E 30초 이내 | CI 실행 리포트 |
| 명확한 실패 메시지 | 실패 원인이 로그만으로 식별 가능 | 코드 리뷰 |

### 3.2. 금지 사항

. 하드코딩된 대기(sleep 고정값) 사용 금지. 명시적 대기(Explicit Wait) 로 대체한다.
. 프로덕션 계정·실데이터 사용 금지. 반드시 테스트 계정·합성 데이터를 사용한다.
. UI 셀렉터로 CSS 클래스명 의존 금지. `data-testid` 속성을 사용한다.

## 4. 유지보수 책임

### 4.1. 오너십 매트릭스

| 자산 | 작성 | 유지보수 | 장애 대응 |
|---|---|---|---|
| Unit Test | 개발자 | 개발자 | 개발자 |
| API 자동화 | QA | QA | QA |
| E2E 자동화 | QA | QA | QA + 기능 담당 개발자 |
| 성능 자동화 | QA + SRE | SRE | SRE |
| CI 파이프라인 | DevOps | DevOps | DevOps |

### 4.2. 유지보수 SLA

. 기능 변경 반영 : 기능 릴리즈와 동시에 자동화 스크립트 갱신 PR을 함께 머지한다.
. Flaky 수정 : 격리 후 2주 이내에 안정화 또는 TC 재설계를 완료한다.
. 실행 실패 방치 금지 : CI 실패 후 24시간 이내에 원인 분석·수정·재실행을 완료한다.

단, 프로덕션 장애 대응 중인 경우 유지보수 SLA를 일시 보류할 수 있으며, QA 리드에게 서면 통보한다.

---

## 5. 자동화 KPI

QA는 월간 리포트에 아래 지표를 포함한다.

| 지표 | 산식 | 목표 |
|---|---|---|
| 자동화 커버리지 | 자동화 TC / 전체 TC | 60% 이상 |
| 자동화 실행 성공률 | 통과 실행 / 전체 실행 | 95% 이상 |
| 자동화 절감 시간 | (수동 실행 시간 - 자동 실행 시간) × 실행 횟수 | 월 40시간 이상 |
| Flaky TC 비율 | Flaky TC / 자동화 TC | 5% 이하 |

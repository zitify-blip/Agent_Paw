결함·테스트 케이스·개선 과제의 우선순위를 P0~P3 4단계로 분류한다. 우선순위는 "언제까지 처리해야 하는가(SLA)"를 결정한다. 심각도(Sev)는 결함의 영향 크기를 나타내며, 우선순위와 독립적으로 판단한다.

---

## 1. 우선순위 분류 기준

### 1.1. 전체 매트릭스

| 우선순위 | 정의 | 대응 SLA(발견 → 수정 완료) | 비고 |
|---|---|---|---|
| P0 | 즉시 조치. 서비스 운영 불가 상태 | 2시간 이내 | 핫픽스·롤백 병행 검토 |
| P1 | 당일 조치. 주요 기능 동작 불가 또는 우회 불가 | 24시간 이내 | 다음 일일 배포에 반영 |
| P2 | 스프린트 내 조치. 부가 기능 이슈 또는 우회 가능 | 2주 이내 | 다음 스프린트 배포 |
| P3 | 백로그 등록. 편의성·개선 성격 | 다음 Minor 릴리즈까지 | 미처리 시 분기 리뷰에서 재평가 |

### 1.2. 분류 판단 플로우

. 사용자 영향 범위 : 전체 > 전체의 50% > 일부 > 단일 사용자 순으로 P0 → P3 하향 조정한다.
. 우회 경로 존재 여부 : 우회 가능하면 최소 P2로 하향한다. 우회 불가하면 P1 이상으로 상향한다.
. 비즈니스 임팩트 : 금전 손실·법적 리스크 동반 시 P0로 즉시 분류한다.

## 2. SLA 상세

### 2.1. P0 대응 체계

| 단계 | 주체 | 조치 | 시한 |
|---|---|---|---|
| 1. 감지·접수 | System / QA | 장애 알림 전파, 인시던트 티켓 Open | 10분 이내 |
| 2. 1차 대응 | 온콜 개발자 | 롤백 또는 임시 우회 조치 | 30분 이내 |
| 3. 근본 수정 | 담당 개발자 | 핫픽스 코드 작성·배포 | 2시간 이내 |
| 4. 사후 분석 | QA + 개발자 | RCA 보고서 작성 | 당일 내 |

### 2.2. P1~P3 대응

| 우선순위 | 수정 데드라인 | 배포 방식 | 리그레션 범위 |
|---|---|---|---|
| P1 | 24시간 이내 | 일일 핫픽스 배포 | 샌티 리그레션 |
| P2 | 2주 이내 | 정규 스프린트 배포 | 전체 리그레션 |
| P3 | Minor 릴리즈 | 정규 Minor 배포 | 스모크 리그레션 |

## 3. 분류 예시

| 상황 | 우선순위 | 사유 |
|---|---|---|
| 전체 사용자 로그인 불가 | P0 | 서비스 이용 자체가 차단됨 |
| 결제 완료 후 영수증 발송 지연 | P1 | 결제는 성공했으나 주요 기능 미작동 |
| 대시보드 그래프 색상 오류 | P3 | 기능·데이터 정확성에 영향 없음 |
| 특정 브라우저에서 레이아웃 깨짐 | P2 | 우회 가능(다른 브라우저 사용) |

단, P0·P1 분류는 반드시 QA 리드 또는 PM 승인을 거쳐야 한다. 임의 하향 조정을 제한한다.

---

## 4. SLA 위반 시 처리

. SLA 초과 감지 시 System이 Slack `#qa-alert` 채널로 자동 에스컬레이션한다.
. 2회 연속 SLA 위반 시 QA 리드가 근본 원인 리뷰를 주관한다.
. 분기 단위로 SLA 준수율을 집계하여 90% 이상을 유지한다.

DEV·STG·QA·PRD 4개 환경의 용도·데이터·접근 권한을 정의한다. 각 환경은 독립된 네트워크·DB·인증 체계를 유지한다.

---

## 1. 환경 개요

### 1.1. 환경별 목적

| 환경 | 코드 | 목적 | 사용 주체 | 안정성 요구 |
|---|---|---|---|---|
| DEV | E01 | 개발자 단위 개발·디버깅 | 개발자 | 낮음(기능 실험 허용) |
| QA | E02 | QA 기능·리그레션 테스트 | QA | 중간(일일 재배포 허용) |
| STG | E03 | UAT, 릴리즈 전 최종 검증 | PM·현업·QA | 높음(운영 준하 안정성) |
| PRD | E04 | 실제 서비스 운영 | 고객 | 최고(무중단) |

### 1.2. 환경별 데이터·연동 비교

| 구분 | DEV | QA | STG | PRD |
|---|---|---|---|---|
| 데이터 소스 | 개발자 임의 시드 | 고정 시드 + 합성 데이터 | PRD 이관 익명화본 | 실제 운영 데이터 |
| DB 스키마 | 최신(개발 중) | 최신 Release 후보 | PRD 동일 + 예정 변경분 | 안정화 버전 |
| 외부 API | Mock | Sandbox | Sandbox | 실 API |
| 결제 게이트웨이 | 비활성 | Sandbox | Sandbox | 실 과금 |
| SMS·이메일 발송 | 로그만 | 내부 수신함 | 지정 테스트 번호만 | 실 발송 |
| 배포 주기 | 커밋마다(수동) | 일 1회 자동 | 스프린트 종료 시 | 스프린트 종료 + 핫픽스 |

## 2. 접근 권한

### 2.1. 권한 매트릭스

| 역할 | DEV | QA | STG | PRD |
|---|---|---|---|---|
| 개발자 | 전체 | 읽기 | 제한 | 불가 |
| QA | 읽기 | 전체 | 전체 | 읽기 |
| PM | 읽기 | 읽기 | 전체 | 읽기 |
| 현업 | 불가 | 불가 | 사용자 권한 | 사용자 권한 |
| 운영자 | 불가 | 불가 | 읽기 | 전체 |
| 외부 벤더 | 불가 | 불가 | 일시 승인 | 불가 |

### 2.2. 접근 절차

. DEV·QA : System이 사내 VPN 접속 시 자동 허용한다.
. STG·PRD : 2단계 인증(2FA) + JIT(Just-In-Time) 권한 승인이 선행되어야 한다.
. PRD 쓰기 권한 : 운영자 3명 내로 제한하며, 모든 쓰기 작업은 감사 로그에 기록한다.

---

## 3. 환경 운영 규칙

### 3.1. 배포 흐름

DEV → QA → STG → PRD 순서를 준수한다. 환경 건너뛰기를 불가하다.

| 이관 방향 | 승인 주체 | 필수 조건 |
|---|---|---|
| DEV → QA | QA | 스모크 리그레션 통과 |
| QA → STG | QA 리드 | 샌티 리그레션 통과 + P0·P1 결함 0건 |
| STG → PRD | PM + QA 리드 | UAT 사인오프 + 전체 리그레션 통과 |

### 3.2. 환경 초기화

. QA 환경 : System이 매주 월요일 05:00 시드 데이터로 초기화한다.
. STG 환경 : System이 매 스프린트 시작 시 PRD 이관 익명화본으로 갱신한다.
. PRD 환경 : 데이터 초기화 불가. 보존·감사 대상이다.

단, QA 환경 초기화 시 진행 중인 테스트가 있으면 QA 리드 승인 후 초기화를 연기한다.

---

## 4. 장애·점검 규칙

. DEV·QA 장애 : 자동 복구 시도 후 30분 내 미복구 시 SRE에 에스컬레이션한다.
. STG·PRD 장애 : 즉시 온콜 엔지니어에 호출하며 SLA에 따라 대응한다.
. 정기 점검 : PRD는 월 1회 일요일 02:00~04:00, STG는 주 1회 금요일 22:00~23:00 점검 시간을 운영한다.

리그레션 테스트를 스모크·샌티·전체 3단계로 구분하고, 변경 유형별 적용 범위와 실행 주기를 정의한다.

---

## 1. 리그레션 레벨

### 1.1. 레벨별 정의

| 레벨 | 목적 | TC 수(기준) | 실행 시간 | 실행 주체 |
|---|---|---|---|---|
| 스모크 | 빌드가 최소한 동작하는지 확인 | 20~30개 | 10분 이내 | 자동화 |
| 샌티 | 변경 영역 + 인접 영역 동작 확인 | 80~120개 | 1시간 이내 | 자동화 + QA |
| 전체 | 전 모듈 회귀 검증 | 500개 이상 | 8시간 이내 | 자동화(야간 배치) |

### 1.2. 레벨별 포함 범위

| 항목 | 스모크 | 샌티 | 전체 |
|---|---|---|---|
| 로그인·인증 | 포함 | 포함 | 포함 |
| 핵심 CRUD(변경 모듈) | - | 포함 | 포함 |
| 인접 CRUD(참조 모듈) | - | 포함 | 포함 |
| 관리자 기능 | - | - | 포함 |
| 배치·스케줄러 | - | - | 포함 |
| 외부 연동(결제·SMS) | - | 변경 시 포함 | 포함 |
| UI 세부 검증 | - | - | 포함 |

## 2. 실행 주기

### 2.1. 트리거별 실행 레벨

| 트리거 | 실행 레벨 | 주체 | 실패 시 조치 |
|---|---|---|---|
| PR 머지 | 스모크 | CI 자동 실행 | 머지 차단 |
| 일일 빌드(01:00) | 샌티 | CI 자동 실행 | Slack 알림 + 담당자 지정 |
| 릴리즈 후보 빌드 | 전체 | QA 수동 트리거 | 릴리즈 연기 검토 |
| 핫픽스 배포 전 | 스모크 + 변경 영역 샌티 | QA 수동 트리거 | 배포 중단 |
| Minor·Major 배포 전 | 전체 + 수동 UAT | QA + PM | 배포 연기 |

### 2.2. 주기 예시(1주)

. 월~금 매일 01:00 : System이 샌티 리그레션을 자동 실행한다.
. 금요일 22:00 : System이 전체 리그레션을 자동 실행한다.
. 주중 PR 머지 시마다 : System이 스모크 리그레션을 자동 실행한다.

## 3. TC 선정 기준

### 3.1. 스모크 TC 선정

. 사용자 수가 가장 많은 Top 10 기능의 Happy Path 만 선정한다.
. 단일 TC 실행 시간 30초 이하를 기준으로 한다.
. 외부 의존성(결제·SMS)은 Mock으로 대체한다.

### 3.2. 샌티 TC 선정

. 변경된 PR의 파일·모듈을 분석하여 영향 범위를 자동 산출한다.
. 영향 모듈의 전체 TC + 직접 참조 모듈의 Happy Path TC 를 포함한다.
. QA가 주 1회 샌티 스위트를 리뷰하여 중복·불필요 TC를 제거한다.

### 3.3. 전체 TC 선정

. 전체 기능 TC + 경계값 TC + 네거티브 TC를 모두 포함한다.
. 성능·보안 TC는 별도 스위트로 분리하며 Minor 릴리즈 직전에만 실행한다.

---

## 4. 리그레션 실패 처리

### 4.1. 실패 분류

| 실패 유형 | 정의 | 조치 |
|---|---|---|
| True Failure | 실제 결함 | 결함 등록 + 수정 |
| Flaky | 재실행 시 통과(타이밍·환경 이슈) | 원인 분석 + 안정화 PR 등록 |
| Obsolete | 기획 변경 반영 누락 | TC 즉시 갱신 |

### 4.2. Flaky TC 관리

. Flaky 판정 : 최근 10회 실행 중 3회 이상 간헐적 실패한 TC를 Flaky로 분류한다.
. Flaky 격리 : QA가 해당 TC를 샌티 스위트에서 일시 제외하고 안정화 이슈를 등록한다.
. 안정화 SLA : Flaky 격리 후 2주 이내에 원인 수정 또는 TC 재설계를 완료한다.

본 문서는 모든 액션 종료 시점에 실행되는 자가점검(Self-Inspection) 워크플로의 상세 구현 지침이다. CLAUDE.md §7 의 최상위 규약을 풀어 쓴 가이드이며, 점검 방법 풀·회차 진행 로직·오류 분류 기준·보고 양식을 정의한다.

---

## 1. 발동 조건

### 1.1. 실행 시점

. 모든 루트(작업 단위)가 종료되는 직전에 System이 자동 실행한다  
. 루트의 정의 : 사용자 단일 요청에 대해 수행한 모든 액션 트리의 최상위 노드  
. 복수 액션이 병렬로 수행된 경우 → 모든 병렬 분기가 완료된 이후 단일 점검 프로세스로 통합 실행한다  
. 사용자 최종 보고 직전에 실행한다. 점검 결과는 보고에 포함한다

### 1.2. 발동 대상 액션

| 액션 유형 | 점검 대상 여부 |
|---|---|
| 문서 작성·수정 (MD·DOCX) | ✅ 필수 |
| 코드 편집 (스크립트·설정) | ✅ 필수 |
| 파일·폴더 이동·삭제 | ✅ 필수 |
| 빌드 스크립트 실행 | ✅ 필수 |
| 단순 조회(Read·Grep·Glob)만 수행한 액션 | ❌ 생략 가능 |
| 사용자 질의에 대한 답변만 수행한 액션 | ❌ 생략 가능 |

단, 조회 기반 액션이라도 사용자에게 **권장·판단·결론**을 제시한 경우 간이 점검(회차 3회)을 수행한다.

## 2. 회차 진행 로직

### 2.1. 회차 구조

. 1회차 : 점검 방법 A 적용 → 오류 검출 여부 기록  
. 2회차 : 점검 방법 B 적용 (A와 다른 관점) → 오류 검출 여부 기록  
. N회차 : 직전 회차와 **중복되지 않는** 방법 선택  
. 전체 회차를 통틀어 **최소 10가지 이상**의 서로 다른 방법을 동원한다

### 2.2. 종료 판정

| 상황 | 종료 여부 | 비고 |
|---|---|---|
| 오류 검출 실패가 3회 연속 발생 | ✅ 종료 | 무결성으로 판정 |
| 10회 미만이며 오류 검출 지속 | ❌ 계속 | 다음 회차 진행 |
| 10회 초과이며 오류 검출 지속 | ❌ 계속 | 3회 연속 무결성까지 계속 |
| 10회 초과이며 3회 연속 무결성 달성 | ✅ 종료 | 최종 무결성 확보 |

. 10회 상한은 **조기 종료 조건이 아니다**. 오류가 있는 한 무결성 확보까지 무제한 반복한다  
. 3회 연속 무결성 판정은 회차가 끊김 없이 이어져야 한다. 중간 회차에서 오류가 1건이라도 검출되면 카운트를 초기화한다

### 2.3. 회차별 상태 전이

```
[회차 시작]
    ↓
[방법 선택 (직전 회차와 다른 관점)]
    ↓
[점검 실행]
    ↓
  ┌─ 오류 있음 → [오류 분류] → [크리티컬 자동 수정 / 마이너 대기]
  │                                ↓
  │                          [무결성 카운트 = 0]
  │                                ↓
  └─ 오류 없음 → [무결성 카운트 +1]
                                   ↓
                              [카운트 == 3?]
                               ├─ Y → [종료]
                               └─ N → [다음 회차로]
```

## 3. 점검 방법 풀 (17종)

회차마다 아래 풀에서 중복 없이 선택한다. 각 방법은 서로 다른 관점의 검증을 제공한다.

| No | 방법명 | 검증 대상 | 검증 내용 |
|---|---|---|---|
| M01 | 파일 존재성 검증 | 참조 파일 | 문서에서 언급·링크한 파일이 실제로 존재하는지 |
| M02 | 링크 유효성 검증 | MD 링크 | 상대 경로·앵커가 유효한지, 깨진 링크 존재 여부 |
| M03 | 문체 규칙 준수 검증 | 본문 종결 어미 | `~한다/~이다/~된다/~불가하다` 사용, 금지 어미(`~합니다`) 부재 |
| M04 | 주어 명시 검증 | 동작 서술 문장 | System/Server/Client/Admin/User 등 행위 주체 명시 여부 |
| M05 | 수치·코드 병기 검증 | 상태값·수치 | `대기(C02101)`, `1년` 등 코드·단위 병기 여부 |
| M06 | 메타데이터 블록 검증 | 문서 상단 | 문서 버전·작성일·작성자 필드 존재 및 포맷 |
| M07 | 변경이력 반영 검증 | 문서 하단 | 이번 변경분이 변경 이력에 추가되었는지, 버전 일치 여부 |
| M08 | 테이블 구조 검증 | MD 테이블 | 헤더·구분선·컬럼 수 일치, 파이프 개수 일관성 |
| M09 | 파일명 규칙 검증 | 생성·이동 파일 | 버전 구분자 `.` 사용, 언더스코어 금지, 확장자 규칙 |
| M10 | 섹션 번호 체계 검증 | 본문 헤딩 | §N.N. 일관성, 누락·중복 번호 부재 |
| M11 | 중복 콘텐츠 검증 | 전체 문서 | 동일 규칙·내용이 여러 위치에 중복 정의되지 않았는지 |
| M12 | SSOT 위반 검증 | 전체 문서 | 단일 출처 원칙을 위반하는 규칙 정의가 없는지 |
| M13 | 교차 참조 일관성 검증 | cross-reference | 참조한 섹션·파일이 실제로 존재하는지 |
| M14 | 이모지 규칙 준수 검증 | 본문 이모지 | 허용 이모지 목록만 사용, 금지 이모지 부재 |
| M15 | 예외 표기 일관성 검증 | 예외·제약 문장 | `단,` / `※` 기호 사용, 형식 일관성 |
| M16 | 사용자 요청 충족 검증 | 산출물 전체 | 사용자 원 요청의 모든 요건이 빠짐없이 반영되었는지 |
| M17 | 부작용(Side-effect) 검증 | 변경 주변 | 이번 변경이 다른 문서·링크·빌드에 미치는 의도치 않은 영향 |
| M18 | 코드 스타일·린트 검증 | 개발 관점 산출물 | `개발/코딩컨벤션.md`·`네이밍규칙.md` 준수, 린트 경고 0건 |
| M19 | DB 스키마 규칙 검증 | DA 관점 산출물 | `DA/테이블네이밍.md`·`컬럼네이밍.md`·`데이터타입.md` 준수, 감사컬럼 5종 존재 |
| M20 | 테스트 커버리지·레벨 검증 | 개발·QA 관점 산출물 | `개발/테스트작성.md` 커버리지 기준 충족, `QA/테스트레벨.md` 책임 경계 준수 |
| M21 | 디자인 토큰·접근성 일관성 검증 | 디자인 관점 산출물 | `디자인/컬러시스템.md`·`타이포그래피.md` 토큰 사용, `접근성.md` WCAG 2.1 AA 준수 |
| M22 | 관점 간 경계·책임 검증 | 경계 파일(상호 참조) | 개발↔QA↔디자인↔DA 간 중복·누락·충돌 여부 (예: 테스트 책임 / 상태표현↔에러처리 / 변수↔DB 네이밍) |

### 3.1. 방법 선택 우선순위

회차 수가 10회 미만인 경우 아래 순서로 선택한다.

. 1순위 : 구조 무결성 (M01, M02, M08, M13) — 깨질 경우 시스템 장애 발생
. 2순위 : 규칙 준수 (M03, M04, M05, M09, M10) — 문서 품질 핵심
. 3순위 : 일관성 (M06, M07, M11, M12, M15) — 중복·누락 방지
. 4순위 : 완결성 (M14, M16, M17) — 요건 충족 및 부작용 검증
. 5순위 : 관점별 특화 (M18~M22) — 해당 관점 산출물이 변경된 경우에만 적용

10회 초과 시에는 지금까지 오류가 검출된 방법을 우선 재적용한다.

### 3.2. 관점별 방법 매핑

| 변경 대상 관점 | 필수 적용 방법 | 보조 적용 방법 |
|---|---|---|
| 기획 (문서) | M01~M17 중 회차별 선택 | M22 (타 관점 경계 영향 시) |
| 개발 (코드·스크립트) | M01, M02, M17, M18 | M20, M22 |
| 디자인 (토큰·컴포넌트) | M01, M02, M17, M21 | M22 |
| QA (테스트 정책·케이스) | M01, M02, M16, M20 | M22 |
| DA (스키마·마이그레이션) | M01, M02, M17, M19 | M22 |

※ M22(관점 간 경계)는 2개 이상 관점을 동시 변경한 경우 반드시 1회 이상 적용한다.

## 4. 오류 분류 기준

### 4.1. 크리티컬 오류

사용자 개입 없이 System이 즉시 자동 수정한다.

| 유형 | 판정 기준 | 처리 방식 |
|---|---|---|
| 깨진 파일 경로 | M01 실패 | 정확한 경로로 자동 보정 |
| 깨진 링크 | M02 실패 | 실제 존재하는 경로로 보정 또는 제거 |
| 빌드 실패 유발 문법 오류 | MD 테이블 파손, JS 구문 오류 | 문법 교정 후 재검증 |
| 시크릿·인증 정보 노출 | 코드 내 평문 토큰·비밀번호 | 즉시 삭제 및 환경변수 분리 |
| 테이블 구조 파손 | M08 실패로 docx 빌드 실패 위험 | 컬럼 수 보정, 구분선 복구 |
| 데이터 무결성 훼손 | 파일 삭제·덮어쓰기로 인한 정보 손실 | 복원 시도, 불가 시 사용자 통지 |
| 파일명 버전 구분자 언더스코어 | M09 실패 | `.` 로 자동 변경 |
| 사용자 요청 요건 누락 | M16 실패 | 누락분 자동 추가 |

### 4.2. 마이너 오류

사용자 개입을 받아 수정 여부를 결정한다.

| 유형 | 판정 기준 | 처리 방식 |
|---|---|---|
| 문체 규칙 위반 | M03 실패 (`~합니다` 등) | 수정 제안 후 사용자 승인 대기 |
| 주어 누락 | M04 실패 | 주어 후보 제시 후 승인 |
| 수치·코드 병기 누락 | M05 실패 | 코드·단위 후보 제시 후 승인 |
| 메타데이터 필드 누락 | M06 실패 | 기본값 제안 후 승인 |
| 이모지 규칙 위반 | M14 실패 | 허용 이모지로 치환 제안 |
| 예외 표기 불일치 | M15 실패 | `단,` / `※` 통일 제안 |
| 섹션 번호 누락·중복 | M10 실패 | 재번호 제안 |
| 용어 일관성 문제 | 동일 개념 다른 용어 사용 | 통일 제안 |
| 사소한 오탈자 | 맞춤법·띄어쓰기 | 교정 제안 |

### 4.3. 분류 모호 시 처리

. 크리티컬·마이너 경계가 모호한 경우 사용자에게 분류 판단을 요청한다  
. 요청 양식 : `[분류 확인 필요] <오류 설명> — 크리티컬/마이너 중 어느 것으로 처리할까요?`  
. 사용자 응답 전까지 해당 오류는 **보류 상태**로 두고 나머지 점검을 진행한다

## 5. 보고 양식

자가점검 종료 후 사용자에게 아래 항목을 간결하게 보고한다.

```
[자가점검 결과]
. 총 검증 회차 : N회
. 종료 사유 : (3회 연속 무결성 | 10회 초과 후 3회 연속 무결성 달성)
. 적용 방법 : M01, M02, M03, ... (중복 없는 고유 방법 목록)
. 크리티컬 자동 수정 : N건
  - [수정 항목 1] <내용>
  - [수정 항목 2] <내용>
. 마이너 대기 : N건 (사용자 판단 필요)
  - [항목 1] <내용> → 제안 : <수정안>
  - [항목 2] <내용> → 제안 : <수정안>
```

. 크리티컬 자동 수정 내역은 반드시 전부 나열한다  
. 마이너 대기 건은 각각 수정 제안을 함께 제시하여 사용자 결정을 돕는다  
. 자가점검 자체에 오류가 발생한 경우(예: 검증 방법 실행 불가) 즉시 사용자에게 알린다

## 6. 예외 처리 및 제약

### 6.1. 점검 면제

. 사용자가 명시적으로 `자가점검 생략` 을 요청한 경우 해당 액션에 한해 면제한다  
. 단, 면제는 해당 요청 범위 내에서만 적용되며 이후 액션에는 기본값(실행)으로 복귀한다

### 6.2. 무한 루프 방지

. 동일 오류가 5회 연속 동일 회차에서 재검출되는 경우 → 자동 수정 불가로 판정하고 사용자 개입을 요청한다  
. 자가점검 총 소요 회차가 **30회를 초과**하는 경우 → 사용자에게 상태를 보고하고 계속 진행 여부를 확인한다

### 6.3. 수정 후 재검증 범위

. 크리티컬 자동 수정 후 → 수정된 파일·섹션에 한정하여 해당 오류 유형만 재검증한다  
. 전체 재검증은 **다음 회차**에서 새로운 방법을 적용할 때 수행한다

## 7. 실행 예시

### 7.1. 정상 케이스 (5회차 조기 종료)

```
1회차 (M01 파일 존재성) → 오류 0건 / 무결성 카운트 1
2회차 (M03 문체 규칙)   → 오류 0건 / 무결성 카운트 2
3회차 (M08 테이블 구조) → 오류 0건 / 무결성 카운트 3 → 종료
```

### 7.2. 오류 검출 케이스 (12회차까지 진행)

```

```

DBA는 본 규칙에 따라 테이블명을 부여한다. 네이밍은 데이터의 성격(마스터·트랜잭션·로그)을 접두어로 식별 가능하도록 설계한다.

---

## 1. 기본 형식

DBA는 모든 테이블명을 소문자 스네이크 케이스(snake_case)로 작성한다. 시스템은 대소문자 구분 이슈를 회피하기 위해 대문자·카멜 케이스 사용을 금지한다.

| 기준 | 규칙 |
|---|---|
| 문자 집합 | 소문자 영문, 숫자, 언더스코어(_)만 허용한다 |
| 단어 구분 | 언더스코어(_)로 분리한다 |
| 최대 길이 | 48자 이하로 제한한다 (Oracle 30자 호환성 여유 확보 목적이 아닌 경우) |
| 언어 | 영문만 사용한다. 한글·숫자 단독 시작 불가하다 |

```
❌  UserOrder, USER_ORDER, user-order, 주문정보
✅  trx_user_order
```

---

## 2. 접두어 체계

데이터 성격에 따라 아래 접두어 중 하나를 부여한다.

| 접두어 | 용도 | 예시 |
|---|---|---|
| TB_ | 일반 엔티티 테이블(분류 애매 시 기본값) | tb_user_profile |
| MST_ | 마스터·기준 정보(거의 변하지 않음) | mst_company, mst_branch |
| TRX_ | 트랜잭션·업무 거래 데이터 | trx_order, trx_payment |
| LOG_ | 로그·이력·감사 기록 | log_login_history, log_api_access |
| CODE_ | 공통 코드·코드 분류 | code_common, code_category |
| TMP_ | 임시·배치 중간 산출 | tmp_daily_aggregate |
| VW_ | 뷰(View) | vw_active_user |

단, MST_/TRX_/LOG_ 접두어는 데이터 특성이 명확한 경우에만 사용하고 판단이 애매할 시 TB_를 부여한다.

---

## 3. 단수·복수 원칙

DBA는 테이블명에 단수형을 사용한다. 테이블은 개념(엔티티)을 표현하며 복수형은 컬렉션 의미를 혼동시킨다.

| 구분 | 단수(사용) | 복수(금지) |
|---|---|---|
| 사용자 | trx_user | trx_users |
| 주문 | trx_order | trx_orders |
| 상품 | mst_product | mst_products |

단, 관계 테이블(다대다 매핑)에서 `_map` 또는 `_rel` 접미어를 사용할 경우 단수로 작성한다. 예) `trx_user_role_map`.

---

## 4. 도메인 접두·접미 패턴

System이 동일 도메인의 테이블을 그루핑할 수 있도록 접두어 이후에 도메인명을 배치한다.

| 패턴 | 예시 | 설명 |
|---|---|---|
| [접두]_[도메인]_[세부] | trx_order_item | order 도메인의 item 하위 엔티티 |
| [접두]_[도메인]_[세부]_map | trx_user_role_map | 다대다 매핑 테이블 |
| [접두]_[도메인]_hist | log_order_hist | 주문 이력(SCD Type 2) |
| [접두]_[도메인]_snapshot | log_order_snapshot | 특정 시점 스냅샷 |

---

## 5. 금지 사항

. 예약어 사용 금지 : USER, ORDER, GROUP 등 SQL 예약어를 테이블명으로 직접 사용 불가하다. 도메인 접두를 조합한다. (예: `trx_user_order`)
. 의미 불명확 축약 금지 : `tbl1`, `data_a` 등 의미를 추론 불가한 명명을 금지한다.
. 시스템 접두 중복 금지 : `tb_log_`, `trx_mst_` 등 접두 중복 조합을 금지한다.
. 날짜·버전 포함 금지 : `trx_order_202604`, `mst_product_v2` 등 시점·버전을 테이블명에 포함 불가하다. 파티셔닝·뷰·스키마 분리로 해결한다.

---

## 6. 안티패턴 및 권장 형태

```
❌  Users, TBL_001, order_202604, user-profile
✅  trx_user, trx_order, mst_product, log_login_history
```

※ 기존 레거시 테이블이 본 규칙과 불일치하는 경우 즉시 리네이밍을 강제하지 않는다. 신규 테이블부터 적용하고, 대규모 스키마 마이그레이션 시 `마이그레이션규칙.md`에 따라 일괄 정비한다.

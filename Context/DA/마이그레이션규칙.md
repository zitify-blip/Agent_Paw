DBA는 운영 환경 스키마 변경을 무중단 원칙에 따라 단계적으로 수행한다. ADD 우선, DROP 분리, 롤백 스크립트 동반을 필수로 한다.

---

## 1. 스키마 변경 절차

Server는 스키마 변경을 아래 6단계로 순차 진행한다.

| 단계 | 담당 | 산출물 |
|---|---|---|
| 1. 변경 설계 | DBA | ERD, DDL 초안, 영향 분석서 |
| 2. 리뷰 | DBA + 개발자 + 운영 | 승인 기록 |
| 3. 개발·스테이징 적용 | 개발자 | 마이그레이션 스크립트, 테스트 결과 |
| 4. 롤백 스크립트 작성 | DBA | rollback.sql |
| 5. 운영 배포(무중단 원칙) | 운영 | 배포 로그 |
| 6. 적용 후 검증 | 개발자 + 운영 | 회귀 테스트 결과 |

---

## 2. 무중단 배포 원칙

DBA는 스키마 변경을 "ADD 먼저, DROP은 별도 배포"의 2단계 이상으로 분리한다. 단일 배포에 ADD·DROP을 혼재하면 롤백 시점의 구버전 애플리케이션이 구조 불일치로 오동작한다.

| 변경 유형 | 배포 분리 방식 |
|---|---|
| 컬럼 추가(ADD COLUMN) | 1회 배포. NULL 허용 또는 DEFAULT 포함 |
| 컬럼 이름 변경 | 1) 신규 컬럼 ADD → 데이터 복사 → 2) 구 컬럼 DROP 별도 배포 |
| 컬럼 타입 변경 | 1) 신규 컬럼 ADD → 데이터 이관 → 2) 애플리케이션 전환 → 3) 구 컬럼 DROP |
| 컬럼 삭제 | 1) 애플리케이션에서 참조 제거 배포 → 2) 컬럼 DROP 별도 배포 |
| 컬럼 NOT NULL 추가 | 1) 기존 데이터 UPDATE(NULL 제거) → 2) NOT NULL 제약 추가 |
| 인덱스 추가 | ONLINE 옵션 적용 (MySQL `ALGORITHM=INPLACE, LOCK=NONE`) |
| 테이블 이름 변경 | 1) 뷰 alias 생성 → 2) 애플리케이션 전환 → 3) 구 테이블 DROP |

단, 대용량 테이블(1억 행 이상)의 ALTER는 pt-online-schema-change 또는 gh-ost 도구 적용을 검토한다.

## 3. 마이그레이션 스크립트 표준

DBA는 마이그레이션 파일을 아래 네이밍 규칙으로 버전 관리한다.

```
V[YYYYMMDDHHMM]__[변경 요약].sql
V202604161030__add_order_memo_column.sql
V202604161100__create_trx_payment_table.sql
V202604161200__drop_legacy_order_no_column.sql
```

| 요소 | 규칙 |
|---|---|
| 접두 | `V` (Flyway 호환) 또는 `U` (Undo) |
| 타임스탬프 | 분 단위까지 기재. 배포 순서 보장 |
| 구분자 | 더블 언더스코어(`__`) |
| 변경 요약 | 소문자 스네이크, 동사 선행(add/create/drop/alter) |

---

## 4. 롤백 스크립트

모든 마이그레이션 파일은 동일 이름의 롤백 스크립트를 짝으로 작성한다.

| 마이그레이션 | 롤백 |
|---|---|
| ADD COLUMN | DROP COLUMN |
| CREATE TABLE | DROP TABLE |
| CREATE INDEX | DROP INDEX |
| 데이터 INSERT | DELETE (WHERE 조건 동일) |
| 데이터 UPDATE | 역방향 UPDATE (이전값 보관 필수) |

```
-- V202604161030__add_order_memo_column.sql
ALTER TABLE trx_order ADD COLUMN memo VARCHAR(500) NULL;

-- U202604161030__add_order_memo_column.sql  (롤백)
ALTER TABLE trx_order DROP COLUMN memo;
```

단, 데이터 UPDATE·DELETE 롤백은 사전 백업을 전제로 한다. 원복 불가능한 변경(컬럼 DROP, TRUNCATE)은 배포 전 전체 백업 스냅샷을 강제한다.

## 5. 배포 시 락(Lock) 관리

Database는 ALTER 구문이 테이블 락을 유발하는지 사전 확인한다.

| DBMS | 무중단 옵션 | 락 회피 방법 |
|---|---|---|
| MySQL 8.0 InnoDB | `ALGORITHM=INPLACE, LOCK=NONE` | 지원 DDL 목록 확인 필수 |
| PostgreSQL | `CREATE INDEX CONCURRENTLY` | 트랜잭션 외부 실행 |
| Oracle | Online Redefinition (DBMS_REDEFINITION) | 별도 절차 준수 |
| SQL Server | `WITH (ONLINE = ON)` | Enterprise 에디션 한정 |

운영 DB의 ALTER 소요 시간이 5분 초과 예상 시 스테이징에서 사전 측정하고 온라인 스키마 변경 도구(pt-osc, gh-ost) 적용을 의무화한다.

## 6. 배포 체크리스트

. 변경 DDL이 CLAUDE.md §4.5 CRUD 제약조건 명세와 일치하는지 확인한다
. 신규 컬럼에 감사 컬럼(`created_at` 등) 포함 여부를 확인한다
. FK 대상 컬럼에 인덱스가 존재하는지 확인한다
. 롤백 스크립트가 짝으로 작성되었는지 확인한다
. 무중단 배포 원칙(ADD 우선·DROP 분리)을 준수하는지 확인한다
. 배포 전 스테이징에서 소요 시간을 측정한다
. 대용량 테이블 ALTER는 온라인 스키마 변경 도구를 적용한다
. 배포 시각은 트래픽 최저점(KST 03:00~05:00)에 수행한다

---

## 7. 금지 사항

. 운영 DB에 직접 DDL 실행을 금지한다. 반드시 마이그레이션 스크립트·형상관리 경로를 거친다.
. 단일 배포에 컬럼 ADD와 DROP 혼재를 금지한다.
. 롤백 스크립트 없는 마이그레이션 배포를 금지한다.
. `DROP TABLE`·`TRUNCATE` 배포는 이해관계자 최종 승인 없이 수행 불가하다.

DBA는 시스템에서 반복 사용되는 분류·상태·구분값을 공통 코드 테이블로 관리한다. 하드코딩·ENUM 대신 중앙 집중 관리하여 유지보수성과 다국어 확장을 확보한다.

---

## 1. 공통 코드 테이블 구조

System은 대분류-중분류-소분류 3단계 계층으로 코드를 관리한다.

| 계층 | 테이블 | 예시 |
|---|---|---|
| 대분류 | code_group | 'ORDER_STATUS' (주문 상태 그룹) |
| 중분류 | code_category | 'PAYMENT_TYPE' (결제 유형) |
| 소분류 | code_common | 'C02101' (대기), 'C02102' (승인) |

단, 규모가 작은 시스템은 대분류·중분류를 합쳐 2단계(group → code)로 운영 가능하다.

---

## 2. 소분류 테이블(code_common) 규격

```
| 컬럼명 | 타입 | 길이 | 필수 | 유니크 | C | U | D | 제약 / 비고 |
|---|---|---|---|---|---|---|---|---|
| code_id | BIGINT | 20 | Y | Y | 자동채번 | ❌ | ❌ | PK |
| group_cd | VARCHAR | 30 | Y | - | ✅ | ❌ | ❌ | 대분류 그룹 코드. code_group 참조 |
| code_value | VARCHAR | 10 | Y | Y(group_cd+code_value) | ✅ | ❌ | ❌ | 코드값. 예) C02101 |
| code_nm | VARCHAR | 100 | Y | - | ✅ | ✅ | ❌ | 코드 표시명. 예) 대기 |
| code_desc | VARCHAR | 500 | N | - | ✅ | ✅ | ❌ | 코드 상세 설명 |
| sort_order | INT | - | Y | - | ✅(기본 0) | ✅ | ❌ | UI 정렬 순서 |
| is_active | CHAR | 1 | Y | - | ✅(기본 'Y') | ✅ | ❌ | 사용 여부. 'Y'/'N' |
| attr1_value | VARCHAR | 100 | N | - | ✅ | ✅ | ❌ | 확장 속성 1 (색상 코드 등) |
| attr2_value | VARCHAR | 100 | N | - | ✅ | ✅ | ❌ | 확장 속성 2 |
```

. 감사 컬럼 5종은 본 명세에서 생략하였으나 실제 테이블에는 반드시 포함한다. (`감사컬럼.md` 참조)

## 3. 코드값 명명 규칙

Server는 코드값을 `[접두문자][대분류번호][중분류번호][순번]` 형식으로 부여한다.

| 구성 요소 | 자릿수 | 예시 |
|---|---|---|
| 접두 문자 | 1자 | C (Common code) |
| 대분류 번호 | 2자리 | 02 |
| 중분류 번호 | 2자리 | 10 |
| 순번 | 2자리 | 01 |
| 전체 | 7자리 | C02101 (7자리인 경우 자릿수 조정) |

표준 형식 : `C` + `XX`(대분류) + `YY`(중분류) + `ZZ`(순번) = 총 7자, 고정 CHAR(7) 또는 VARCHAR(10).

```
예시 체계:
- 대분류 02 = 주문(ORDER)
- 중분류 0210 = 주문 상태(ORDER_STATUS)
- 소분류 :
    C021001 : 대기
    C021002 : 승인
    C021003 : 반려
    C021099 : 기타
```

단, 기존 시스템 호환이 필요한 경우 `C02101`(6자리) 관례를 유지하고 자릿수를 시스템 전반에 일관되게 유지한다.

## 4. 코드 그룹별 상태값 정의 예시

```
## N.N. 주문 상태 코드 (group_cd = 'ORDER_STATUS')

. 대기(C021001) : User가 주문 등록을 완료했으나 결제가 미완료된 상태이다.
. 승인(C021002) : System이 결제 승인을 확인한 상태이다.
. 배송중(C021003) : 배송사가 상품을 집화한 이후 상태이다.
. 완료(C021004) : User가 수령 확인 또는 자동 확정된 상태이다.
. 취소(C021098) : User 또는 Admin이 주문을 취소한 상태이다.
. 기타(C021099) : 예외 분기. 사유 별도 기록한다.
```

## 5. 운영 원칙

| 원칙 | 설명 |
|---|---|
| 삭제 금지 | 한 번 부여된 코드값은 삭제 불가하다. `is_active = 'N'`으로 비활성화한다 |
| 재사용 금지 | 비활성화된 코드값을 다른 의미로 재부여 불가하다 |
| 하드코딩 금지 | Application Layer는 코드값(C021001) 상수를 enum·상수 클래스로만 참조한다. 문자열 직접 기재를 금지한다 |
| 캐시 운영 | Server는 code_common 전체를 메모리 캐시(TTL 10분)로 로드하여 조회한다 |

---

## 6. FK 참조 방식

Server는 업무 테이블의 코드값 컬럼(`status_cd`)에 FK 제약을 선언하지 않는 것을 표준으로 한다.

| 방식 | 장점 | 단점 |
|---|---|---|
| FK 선언 | 무결성 강제 | 코드 추가 시 배포 순서 제약, 캐시 갱신 복잡도 |
| FK 미선언(Application Layer 검증) | 유연한 배포, 캐시 활용 | 불일치 발생 가능 |

DBA는 FK 미선언 방식을 기본으로 하고, Application Layer에서 코드 캐시 기반 검증을 수행한다.

---

## 7. 금지 사항

. 업무 테이블에 코드명을 문자열로 중복 저장하는 것을 금지한다. `status_nm = '대기'`를 함께 저장하면 다국어·표시명 변경 시 전체 UPDATE가 필요하다.
. ENUM 타입 사용을 금지한다. 값 추가 시 ALTER TABLE이 요구되어 무중단 배포에 제약이 발생한다.

DBA는 데이터 삭제 시 논리 삭제(Soft Delete)와 물리 삭제(Hard Delete)를 데이터 성격·복구 요구·감사 요구를 기준으로 구분 적용한다.

---

## 1. 논리 삭제 vs 물리 삭제

System은 아래 기준에 따라 삭제 방식을 선택한다.

| 구분 | 논리 삭제 | 물리 삭제 |
|---|---|---|
| 정의 | deleted_at 컬럼 설정, 실제 행은 유지 | DELETE 문으로 행 자체 제거 |
| 복구 | UPDATE deleted_at = NULL로 환원 가능 | 백업 복원 외 복구 불가 |
| 감사 | 삭제 이력 DB에 보존 | 별도 감사 로그 없으면 추적 불가 |
| 저장 비용 | 누적 증가 | 즉시 회수 |
| 조회 성능 | deleted_at 필터 강제, 인덱스 영향 | 무관 |
| 권장 대상 | 업무 트랜잭션·마스터(주문·회원·상품) | 로그·임시·대용량 이력(파기 대상) |

---

## 2. 논리 삭제 적용 기준

DBA는 아래 조건 중 하나 이상 충족 시 논리 삭제를 채택한다.

. 법령·규정상 보관 의무가 있는 데이터
. 회계·정산 대상 거래 데이터
. FK로 참조되는 마스터 데이터 (주문이 참조하는 상품 등)
. 감사 추적이 필요한 관리자 조작 대상 데이터

단, 1일 100만 건 이상 발생하는 로그 테이블은 논리 삭제 대신 파티셔닝 기반 물리 파기를 적용한다.

---

## 3. deleted_at 컬럼 규격

```
| 컬럼명 | 타입 | 길이 | 필수 | 유니크 | C | U | D | 제약 / 비고 |
|---|---|---|---|---|---|---|---|---|
| deleted_at | TIMESTAMP | - | N | - | ❌ | ✅(시스템) | ❌ | 논리 삭제 시 System이 UTC 일시를 설정한다. NULL이면 활성 |
| deleted_by | BIGINT | 20 | N | - | ❌ | ✅(시스템) | ❌ | 삭제 처리 User의 user_id |
```

. deleted_at이 NULL → 활성(Active) 레코드
. deleted_at이 NOT NULL → 삭제됨(Deleted). 조회 대상에서 제외

---

## 4. 조회 뷰 표준화

DBA는 논리 삭제 필터 누락을 방지하기 위해 활성 레코드만 노출하는 뷰를 표준 제공한다.

| 뷰 이름 | 용도 | WHERE 조건 |
|---|---|---|
| vw_[도메인]_active | 일반 조회 | deleted_at IS NULL |
| vw_[도메인]_all | 관리자 조회(삭제 포함) | 조건 없음 |

```
CREATE VIEW vw_order_active AS
SELECT * FROM trx_order
WHERE deleted_at IS NULL;
```

단, Application Layer에서 직접 테이블을 조회할 때도 반드시 `WHERE deleted_at IS NULL` 조건을 포함한다. 공통 Repository 계층에 글로벌 필터를 강제하는 방안을 권장한다.

---

## 5. 인덱스 고려사항

Database는 논리 삭제 테이블에 아래 인덱스 전략을 적용한다.

| 전략 | 설명 | 적용 DBMS |
|---|---|---|
| 복합 인덱스 후행 배치 | `idx_order_status (status_cd, deleted_at)` | 모든 DBMS |
| Partial Index | `CREATE INDEX ... WHERE deleted_at IS NULL` | PostgreSQL |
| Filtered Index | `CREATE INDEX ... WHERE deleted_at IS NULL` | SQL Server |
| Generated Column | `is_active AS (deleted_at IS NULL)` + 인덱싱 | MySQL 5.7+ |

활성 레코드 비율이 낮을수록(삭제 비율 높을수록) Partial/Filtered Index 효과가 크다.

---

## 6. 유니크 제약 처리

논리 삭제 시 동일 값의 재등록을 허용하려면 유니크 제약을 `deleted_at` 조합으로 확장한다.

```
-- ❌ 논리 삭제된 이메일의 재등록 불가
UNIQUE KEY uk_user_email (email)

-- ✅ 활성 레코드만 유니크 (삭제된 이메일은 재사용 가능)
UNIQUE KEY uk_user_email_active (email, deleted_at)
-- 또는 PostgreSQL : CREATE UNIQUE INDEX ... WHERE deleted_at IS NULL
```

단, 개인정보 재등록 시 이전 이력 승계 정책(회원 탈퇴 후 재가입)에 따라 유니크 전략을 재검토한다.

---

## 7. 물리 파기 스케줄

Server는 논리 삭제된 레코드를 영구 파기하는 배치를 아래 주기로 운영한다.

| 데이터 유형 | 논리 삭제 보존 | 물리 파기 시점 |
|---|---|---|
| 회원 정보 | 1년 (개인정보보호법 준수) | 1년 경과 후 마스킹/파기 |
| 주문 정보 | 5년 | 5년 경과 후 아카이브 이관 |
| 일반 업무 데이터 | 2년 | 2년 경과 후 파기 |

---

## 8. 금지 사항

. `is_deleted` 플래그(CHAR(1)) 단독 사용을 지양한다. 삭제 시점 추적이 불가하다. `deleted_at` TIMESTAMP를 표준으로 사용한다.
. 논리 삭제 없이 물리 삭제만 허용하는 FK 참조 테이블은 CASCADE 연쇄 삭제 위험을 유발하므로 금지한다.

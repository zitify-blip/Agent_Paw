DBA는 데이터의 시간 축 변경을 보존하기 위해 SCD(Slowly Changing Dimension) 전략을 선정한다. 이력 보존 수준과 조회 패턴을 기반으로 Type 1/2/3 중 적합한 방식을 채택한다.

---

## 1. SCD Type 비교

데이터의 변경 이력을 어떻게 저장할지를 아래 3가지 전략으로 구분한다.

| 구분 | SCD Type 1 | SCD Type 2 | SCD Type 3 |
|---|---|---|---|
| 정의 | 이력 미보존, 덮어쓰기 | 이력 행 추가, 버전별 보존 | 이전 값 1회만 컬럼 보존 |
| 과거값 조회 | 불가 | 전체 이력 조회 가능 | 직전 값만 조회 가능 |
| 저장 구조 | 단일 테이블 갱신 | 이력 테이블 별도(또는 동일 테이블 내 버전 컬럼) | 컬럼 2쌍(current/previous) |
| 저장 비용 | 최소 | 최대 | 중간 |
| 권장 대상 | 갱신 빈도 낮고 이력 불필요(예: 사용자 닉네임) | 가격·계약 조건·권한 이력 | 부서 이동·담당자 교체(직전만 필요) |

---

## 2. SCD Type 2 이력 테이블 설계

DBA는 Type 2를 본 테이블과 이력 테이블로 분리한 "Shadow History" 구조로 설계한다.

| 항목 | 본 테이블 (mst_product) | 이력 테이블 (log_product_hist) |
|---|---|---|
| 용도 | 현재 유효한 1건 | 변경 시점별 스냅샷 N건 |
| PK | product_id | hist_id (자동채번) |
| 이력 식별 | - | product_id + valid_from_at |
| 유효 기간 컬럼 | - | valid_from_at, valid_to_at |
| 현재 레코드 표시 | - | is_current (CHAR(1)) |
| 변경 사유 | - | change_reason |

---

## 3. 이력 테이블 표준 컬럼

Server는 Type 2 이력 테이블에 아래 컬럼을 반드시 포함한다.

```
| 컬럼명 | 타입 | 길이 | 필수 | 유니크 | C | U | D | 제약 / 비고 |
|---|---|---|---|---|---|---|---|---|
| hist_id | BIGINT | 20 | Y | Y | 자동채번 | ❌ | ❌ | 이력 PK |
| product_id | BIGINT | 20 | Y | - | ✅ | ❌ | ❌ | 본 테이블 참조 |
| valid_from_at | TIMESTAMP | - | Y | - | ✅ | ❌ | ❌ | 이 버전이 유효해진 시점(UTC) |
| valid_to_at | TIMESTAMP | - | N | - | ✅ | ✅(시스템) | ❌ | 이 버전이 만료된 시점. NULL이면 현재 유효 |
| is_current | CHAR | 1 | Y | - | ✅(기본 'Y') | ✅(시스템) | ❌ | 'Y'=현재, 'N'=과거 이력 |
| change_reason_cd | VARCHAR | 10 | N | - | ✅ | ❌ | ❌ | 변경 사유 코드 |
| [본 테이블 업무 컬럼 전체 복제] | | | | | | | | 스냅샷 값 저장 |
```

---

## 4. 이력 적재 방식

DBA는 이력 적재 주기·시점을 업무 특성에 맞춰 선정한다.

| 방식 | 트리거 | 장점 | 단점 |
|---|---|---|---|
| DB 트리거 | UPDATE 발생 시 DB가 자동 INSERT | 누락 방지, 애플리케이션 투명 | 디버깅 난이도 증가, 성능 부담 |
| Application 레이어 | Server가 명시적 INSERT | 제어·테스트 용이 | 개발자 누락 위험 |
| CDC(Change Data Capture) | Binlog·WAL 기반 스트리밍 | 실시간 이력, 외부 분석계 연동 가능 | 인프라 비용, 운영 복잡도 |
| 배치 스냅샷 | 일일 00:00 KST 전체 복제 | 구현 단순 | 일 단위 이하 이력 유실 |

단, 감사(Audit) 목적 이력은 반드시 DB 트리거 또는 CDC로 구현한다. Application Layer는 버그·우회 경로로 누락 가능성이 존재한다.

## 5. 이력 조회 패턴

```
-- 현재 유효한 값
SELECT * FROM log_product_hist
WHERE product_id = 100 AND is_current = 'Y';

-- 특정 시점의 값
SELECT * FROM log_product_hist
WHERE product_id = 100
  AND valid_from_at <= '2026-03-01 00:00:00'
  AND (valid_to_at > '2026-03-01 00:00:00' OR valid_to_at IS NULL);

-- 전체 변경 이력 순차 조회
SELECT * FROM log_product_hist
WHERE product_id = 100
ORDER BY valid_from_at;
```

---

## 6. 이력 보존 기간

. 일반 업무 이력 : 5년 보관
. 금융·정산 이력 : 10년 보관
. 개인정보 관련 이력 : 관련 법령(개인정보보호법)에 따른 보관 기간 준수

단, 보관 기간 경과 후 아카이브 스토리지(S3·Cold Storage)로 이관하고 운영 DB에서는 파기한다.

---

## 7. 금지 사항

. 본 테이블에 `version` 컬럼 증가 방식으로 이력을 섞어 저장하는 것을 권장하지 않는다. 현재값 조회 시 항상 MAX(version) 필터가 필요해 성능 저하를 유발한다.
. 이력 테이블의 PK를 본 테이블 PK로 재사용하는 것을 금지한다. 같은 product_id의 이력이 다수 존재해 PK 유일성을 파괴한다.

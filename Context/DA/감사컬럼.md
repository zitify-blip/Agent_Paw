DBA는 모든 업무 테이블(MST_/TRX_)에 표준 감사 컬럼 5종을 강제 포함한다. 감사 컬럼은 데이터 생성·수정·삭제 시점 및 주체 추적을 목적으로 한다.

---

## 1. 표준 감사 컬럼 5종

System은 아래 5개 컬럼을 모든 업무 테이블에 일관된 형태로 배치한다.

| 컬럼명 | 타입 | 필수 | 설명 |
|---|---|---|---|
| created_at | TIMESTAMP | Y | 레코드 최초 생성 일시(UTC). DEFAULT CURRENT_TIMESTAMP |
| created_by | BIGINT(20) | Y | 생성한 User의 user_id. 시스템 자동 INSERT 시 0 또는 시스템 전용 ID |
| updated_at | TIMESTAMP | Y | 레코드 최종 수정 일시(UTC). ON UPDATE CURRENT_TIMESTAMP |
| updated_by | BIGINT(20) | Y | 최종 수정한 User의 user_id |
| deleted_at | TIMESTAMP | N | 논리 삭제 일시(UTC). NULL이면 활성 레코드 |

단, LOG_/TMP_ 접두 테이블은 `updated_*`, `deleted_at` 컬럼을 생략 가능하다. 로그는 변경·삭제 대상이 아니기 때문이다.

## 2. CRUD 제약조건 명세

```
| 컬럼명 | 타입 | 길이 | 필수 | 유니크 | C | U | D | 제약 / 비고 |
|---|---|---|---|---|---|---|---|---|
| created_at | TIMESTAMP | - | Y | - | ✅(기본 CURRENT_TIMESTAMP) | ❌ | ❌ | INSERT 시 System이 자동 기록한다. 변경 불가하다 |
| created_by | BIGINT | 20 | Y | - | ✅(시스템) | ❌ | ❌ | 세션 User의 user_id. 시스템 작업은 0 고정 |
| updated_at | TIMESTAMP | - | Y | - | ✅(기본 CURRENT_TIMESTAMP) | ✅(시스템) | ❌ | UPDATE 시 System이 자동 갱신한다 |
| updated_by | BIGINT | 20 | Y | - | ✅(시스템) | ✅(시스템) | ❌ | 최종 수정 세션 User의 user_id |
| deleted_at | TIMESTAMP | - | N | - | ❌ | ✅(시스템) | ❌ | 논리 삭제 시 System이 UTC 일시를 설정한다. 복구 시 NULL로 환원 |
```

## 3. 컬럼 배치 순서

DBA는 모든 테이블 마지막에 감사 컬럼을 아래 순서로 배치한다. 테이블 간 일관성이 개발자 가독성을 높인다.

```
CREATE TABLE trx_order (
  order_id       BIGINT(20)   NOT NULL AUTO_INCREMENT,
  user_id        BIGINT(20)   NOT NULL,
  order_no       VARCHAR(20)  NOT NULL,
  status_cd      CHAR(6)      NOT NULL,
  total_amt      DECIMAL(15,2) NOT NULL,
  -- 업무 컬럼 종료 --
  created_at     TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
  created_by     BIGINT(20)   NOT NULL,
  updated_at     TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  updated_by     BIGINT(20)   NOT NULL,
  deleted_at     TIMESTAMP    NULL,
  PRIMARY KEY (order_id)
);
```

## 4. 기록 주체 규칙

Server는 감사 컬럼의 `*_by` 필드에 현재 세션의 User ID를 기록한다.

| 시나리오 | created_by / updated_by |
|---|---|
| 일반 User의 화면 조작 | 해당 User의 user_id |
| Admin의 관리 조작 | 해당 Admin의 user_id |
| 시스템 배치 작업 | 0 (또는 예약 시스템 계정 user_id) |
| 외부 연동 API 호출 | 연동 계정의 user_id |
| 익명 사용자(로그인 전) | -1 (예약 익명 ID) |

단, 시스템 계정 ID는 운영 정책에 따라 `mst_user`에 사전 INSERT하고 고정 user_id를 부여한다.

---

## 5. 타임존 규칙

. 저장 타임존 : 모든 감사 TIMESTAMP는 UTC 기준으로 저장한다.
. 표시 타임존 : Client는 UTC+9(KST)로 변환하여 노출한다.
. 서버 세션 타임존 : Database 세션은 `+00:00`으로 고정하여 DST·서버 로케일 종속성을 제거한다.

```
❌  created_at = '2026-04-16 14:30:00' (서버 로컬 시간 기록)
✅  created_at = '2026-04-16 05:30:00 UTC' → Client 표시 '2026-04-16 14:30:00 KST'
```

---

## 6. 금지 사항

. 감사 컬럼 직접 UPDATE 금지 : Application Layer에서 `created_at`, `created_by` 변경을 시도하면 System이 거부한다.
. `created_at` NULL 허용 금지 : 레코드 생성 시점은 반드시 존재한다.
. 감사 컬럼을 업무 조건으로 사용 금지 : 만료 판정 등은 별도 `expire_dt` 컬럼을 도입한다. `updated_at`을 업무 로직에 재활용하면 감사성이 훼손된다.

DBA는 조회 패턴·카디널리티·저장 비용을 종합 분석하여 인덱스를 설계한다. 무분별한 인덱스 생성은 INSERT/UPDATE 성능을 저하시킨다.

---

## 1. 인덱스 선정 기준

DBA는 아래 조건 중 2개 이상을 충족할 때 인덱스 생성을 검토한다.

| 기준 | 임계치 |
|---|---|
| 테이블 행 수 | 1만 행 이상 |
| WHERE 절 사용 빈도 | 주간 100회 이상 쿼리에 등장 |
| 카디널리티(고유값 비율) | 30% 이상 (낮은 중복률) |
| 조회 응답 시간 | 미인덱스 시 100ms 초과 |

단, 10만 행 이하 소규모 테이블은 인덱스 없이 Full Scan이 더 효율적인 경우가 존재한다. EXPLAIN으로 실측 확인 후 결정한다.

---

## 2. 단일 vs 복합 인덱스

Server는 WHERE 절의 조건 컬럼 개수와 조합에 따라 인덱스 구조를 선정한다.

| 구분 | 적용 기준 | 예시 |
|---|---|---|
| 단일 인덱스 | 단일 컬럼 조회 빈도 절대 우세 | `idx_user_email (email)` |
| 복합 인덱스 | 2~3개 컬럼 AND 조건 고정 | `idx_order_user_status (user_id, status_cd)` |
| 복합 인덱스(정렬 포함) | ORDER BY 최적화 목적 | `idx_order_user_created (user_id, created_at DESC)` |

단, 복합 인덱스 컬럼은 4개를 초과하지 않는다. 선행 컬럼부터 조건이 걸려야 인덱스가 적용된다(Left-Prefix Rule).

---

## 3. 복합 인덱스 컬럼 순서

DBA는 복합 인덱스 컬럼을 아래 우선순위로 배치한다.

. 1순위 : 등호(=) 조건 컬럼
. 2순위 : 카디널리티 높은 컬럼 (고유값 많은 순)
. 3순위 : 범위(BETWEEN, >, <) 조건 컬럼
. 4순위 : ORDER BY 컬럼

```
쿼리 : WHERE status_cd = 'C02101' AND user_id = 123 AND created_at > '2026-01-01'
     ORDER BY created_at DESC

❌  idx (created_at, status_cd, user_id)      — 범위 조건이 선두
✅  idx (user_id, status_cd, created_at)      — 등호 → 등호 → 범위+정렬
```

---

## 4. 커버링 인덱스

Database는 쿼리에 필요한 모든 컬럼을 인덱스에 포함하여 테이블 접근(Random I/O)을 제거한다.

| 항목 | 내용 |
|---|---|
| 적용 조건 | SELECT 컬럼 + WHERE + ORDER BY 컬럼 수 합계가 5개 이하 |
| 효과 | 테이블 접근 없이 인덱스만으로 결과 반환 |
| 비용 | 인덱스 크기 증가, INSERT/UPDATE 오버헤드 상승 |

```
쿼리 : SELECT user_id, order_no, total_amt
     FROM trx_order
     WHERE status_cd = 'C02101'

커버링 인덱스 : idx_order_covering (status_cd, user_id, order_no, total_amt)
```

단, BLOB/TEXT 대용량 컬럼은 커버링 인덱스에 포함하지 않는다.

---

## 5. 카디널리티 고려

카디널리티(Cardinality)는 인덱스 컬럼의 고유값 비율을 의미한다. 낮은 카디널리티 컬럼을 단독 인덱싱하면 효과가 미미하다.

| 카디널리티 | 비율(고유값/전체행) | 단독 인덱스 효과 |
|---|---|---|
| 매우 높음 | 90% 이상 | 최고 (PK, UNIQUE 키 수준) |
| 높음 | 30% 이상 | 효과적 |
| 중간 | 10~30% | 복합 인덱스 후행 컬럼 권장 |
| 낮음 | 10% 미만 | 단독 인덱스 비효율. 선행 컬럼과 결합 |

예) `is_deleted`, `status_cd`(상태 5종 이하) 컬럼은 단독 인덱싱 대신 복합 인덱스 후행에 배치한다.

---

## 6. 인덱스 유형별 사용 지침

| 유형 | 용도 | 비고 |
|---|---|---|
| B-Tree(기본) | 등호·범위 조회, 정렬 | 대부분 DBMS 기본값 |
| Hash | 등호 조회 전용 | Memory 엔진, 범위 조회 불가 |
| Full-Text | 대용량 텍스트 검색 | LIKE '%keyword%' 대체 |
| Spatial | 지리 좌표 | POINT, POLYGON 컬럼 |
| Partial Index | 조건부 인덱스 | PostgreSQL. WHERE deleted_at IS NULL 등 |

---

## 7. 금지 사항

. 동일 컬럼 조합에 중복 인덱스 생성 금지 : `idx_a(col1)`와 `idx_b(col1, col2)` 공존 시 전자는 제거한다.
. 업데이트 빈발 컬럼 단독 인덱싱 금지 : `updated_at` 단독 인덱스는 매 UPDATE마다 인덱스 재작성을 유발한다.
. 함수 기반 쿼리 시 단순 인덱스 무용지물 : `WHERE UPPER(email) = ?`에는 함수 기반 인덱스(Function-based Index)를 별도 생성한다.

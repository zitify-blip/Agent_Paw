DBA는 Primary Key와 Foreign Key를 설계할 때 유일성·참조 무결성·성능·분산 확장성을 종합 고려한다.

---

## 1. PK 생성 방식 비교

DBA는 아래 3종 중 테이블 특성에 적합한 방식을 선정한다.

| 방식 | 타입 | 장점 | 단점 | 권장 용도 |
|---|---|---|---|---|
| 자동채번(AUTO_INCREMENT) | BIGINT(20) | 인덱스 효율 최고, 순차 삽입 | 단일 DB 노드 종속, 값 노출 시 정보 유출 | 일반 트랜잭션·마스터 테이블(기본값) |
| UUID v4 | CHAR(36) | 분산 생성 가능, 충돌 없음 | 인덱스 단편화·저장 공간 2~3배 | 분산 시스템·외부 노출 식별자 |
| 복합 PK | 다중 컬럼 | 자연키 그대로 사용 | 조인 비용 증가, FK 전파 부담 | 매핑 테이블(다대다) |
| 업무 식별자(비즈니스 키) | VARCHAR | 자연 의미 전달 | 변경 불가 전제 필요 | 사용 금지(대리키 사용 원칙) |

DBA는 기본값으로 **BIGINT 자동채번** PK를 사용한다. 외부 노출이 필요한 경우 별도 `public_id` 컬럼(UUID)을 추가하고 PK는 내부 BIGINT를 유지한다.

---

## 2. PK 설계 원칙

. 모든 테이블은 단일 PK를 반드시 보유한다. PK 없는 테이블 생성을 금지한다.
. PK 컬럼명은 `id` 또는 `[도메인]_id`로 통일한다. 예) `user_id`, `order_id`.
. PK 값 변경을 금지한다. 변경이 필요한 업무 식별자는 별도 컬럼으로 분리한다.
. 복합 PK는 매핑 테이블에 한정하며, 일반 엔티티 테이블은 대리키(surrogate key) 단일 PK를 사용한다.

```
❌  CREATE TABLE trx_order (order_no VARCHAR(20) PRIMARY KEY, ...)
    — 업무 식별자를 PK로 사용 (변경 시 전파 비용)

✅  CREATE TABLE trx_order (
      order_id BIGINT(20) AUTO_INCREMENT PRIMARY KEY,
      order_no VARCHAR(20) UNIQUE NOT NULL,
      ...
    )
```

## 3. FK 제약 조건 정책

DBA는 자식 레코드가 부모 삭제/변경 시 어떻게 동작할지 명시적으로 선언한다.

| 정책 | ON DELETE | ON UPDATE | 권장 적용 |
|---|---|---|---|
| RESTRICT (기본 권장) | 자식 존재 시 부모 삭제 차단 | 동일 | 마스터 참조(mst_user, mst_product) |
| CASCADE | 부모 삭제 시 자식 일괄 삭제 | 부모 PK 변경 시 자식 FK 동시 갱신 | 소유 관계(주문-주문상세, 게시글-첨부파일) |
| SET NULL | 부모 삭제 시 자식 FK를 NULL로 설정 | 동일 | 선택 참조(담당자 퇴사 시 담당자 NULL) |
| NO ACTION | RESTRICT와 동일(SQL 표준) | 동일 | DBMS 간 이식성 고려 시 |

단, 대용량 테이블(수천만 행 이상) 간 CASCADE는 Lock 경쟁을 유발한다. Application Layer 삭제 로직으로 대체를 검토한다.

## 4. FK 인덱스 규칙

Database는 FK 컬럼에 반드시 인덱스를 생성한다. 인덱스 미생성 시 부모 테이블 갱신 시 자식 테이블 전체 스캔이 발생한다.

. FK 선언 시 동일 컬럼에 인덱스를 자동 생성하는 DBMS(MySQL InnoDB)와 그렇지 않은 DBMS(PostgreSQL)를 구분하여 명시적 인덱스를 추가한다.
. 복합 인덱스의 선행 컬럼이 FK인 경우 별도 단일 FK 인덱스는 생략 가능하다.

## 5. CRUD 제약조건 예시

```
| 컬럼명 | 타입 | 길이 | 필수 | 유니크 | C | U | D | 제약 / 비고 |
|---|---|---|---|---|---|---|---|---|
| order_id | BIGINT | 20 | Y | Y | 자동채번 | ❌ | ❌ | PK. AUTO_INCREMENT. 변경 불가하다 |
| user_id | BIGINT | 20 | Y | - | ✅ | ✅ | ❌ | FK → mst_user.user_id. ON DELETE RESTRICT |
| order_no | VARCHAR | 20 | Y | Y | ✅(시스템) | ❌ | ❌ | 주문 채번 규칙 별도. 변경 불가하다 |
| assignee_user_id | BIGINT | 20 | N | - | ✅ | ✅ | ❌ | FK → mst_user.user_id. ON DELETE SET NULL |
```

---

## 6. 금지 사항

. 자연키(이메일, 휴대폰, 주민번호)를 PK로 사용하는 것을 금지한다. 변경 가능성 및 개인정보 정책 변경에 취약하다.
. FLOAT/DOUBLE PK 사용을 금지한다. 근사값으로 인한 조회 실패 위험이 존재한다.
. NULL 허용 PK를 금지한다.

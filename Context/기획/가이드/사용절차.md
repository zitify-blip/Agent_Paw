## 1. 새 문서 작성

1. 작성하려는 산출물의 유형에 맞는 템플릿을 선택한다 (`템플릿목록.md` 참조).
2. 템플릿 파일을 산출물 폴더(`Docs/`)에 `파일명규칙.md` 형식으로 복사한다.

   ```bash
   cp Context/템플릿/API명세서.md Docs/ServicePlex_API명세서_v1.0.md
   ```

3. 메타데이터(`공통섹션/메타데이터.md`)를 작성한다. 파일명의 버전과 메타데이터의 버전을 일치시킨다.
4. 각 섹션의 가이드에 따라 내용을 작성한다. 사용하지 않는 선택 섹션은 삭제한다.
5. 변경 이력(`공통섹션/변경이력.md`)에 v1.0 초안 작성 항목을 추가한다.
6. [문체규칙.md](문체규칙.md) · [이모지규칙.md](이모지규칙.md) · [상태표기.md](상태표기.md) · [문서유형별_예외.md](문서유형별_예외.md) · [작문_페르소나.md](작문_페르소나.md) 를 준수했는지 확인한다.

## 2. 버전 업데이트

### Patch (오탈자·토시 수정)

```bash
cp Docs/ServicePlex_API명세서_v1.0.0.md Docs/ServicePlex_API명세서_v1.0.1.md
mv Docs/ServicePlex_API명세서_v1.0.0.md Docs/old/
```

. 메타데이터의 버전과 작성일을 갱신한다.
. 변경 이력에 새 Patch 항목을 추가한다 (수정 섹션 번호 명시).

### Minor (섹션 신설·모듈 추가)

```bash
cp Docs/ServicePlex_API명세서_v1.0.1.md Docs/ServicePlex_API명세서_v1.1.0.md
mv Docs/ServicePlex_API명세서_v1.0.1.md Docs/old/
```

. 메타데이터·변경 이력을 갱신한다.
. 영향 범위(`공통섹션/영향범위.md`)를 재검토한다.

### Major (전면 재작성·구조 개편)

. 새 버전 파일을 작성한다.
. 이전 최신 버전 파일을 `Docs/old/` 로 이동한다 (파일명 유지).
. 변경 이력에 Major 변경 사유를 상세히 기술한다.
. 승인 기록(`공통섹션/승인기록.md`)을 다시 받는다.

## 3. 구버전 보관 규칙

. 보관 위치 : `Docs/old/`. 폴더가 없으면 버전업 시점에 생성한다.
. 이동 시점 : 새 버전 파일을 `Docs/` 에 저장한 직후, 기존 최신 버전을 `Docs/old/` 로 이동한다.
. 파일명 유지 : 이동 시 파일명을 변경하지 않는다.
. 노출 원칙 : `Docs/` 직속에는 각 산출물의 최신 버전 파일 1개만 노출한다.
. 삭제 금지 : 구버전은 감사·회귀 추적 목적으로 영구 보존한다.

(CLAUDE.md §6.4 참조)

## 4. DOCX 출력

. "산출물 docx로 뽑아줘" 요청 시 시스템통합기획서·서비스정책서·사용자매뉴얼 3종을 동시 생성한다.
. DOCX 생성 시 CLAUDE.md §5 "DOCX 표 가독성 지침" 을 준수한다 (열 너비, cantSplit, 헤더 반복, 스트라이프, makeTable 패턴, 특수문자 이스케이프).
. 파일명은 MD 와 동일 규칙(`파일명규칙.md`)으로 생성한다. 버전 표기에 점(`.`)을 사용한다.

## 5. Git 커밋 규칙

```
docs: [동작] [문서명] [버전]

동작: Add / Update / Archive / Fix / Refactor
```

예:

```
docs: Update ServicePlex_시스템통합기획서 v0.6.2 — §15 공급업체 등록 신설
docs: Archive ServicePlex_서비스정책서 v0.4.0 → Docs/old/
```

## 6. 신규 관점(개발·디자인·QA·DA) 산출물 절차

기획 외 4개 관점 폴더(`Context/개발/`, `Context/디자인/`, `Context/QA/`, `Context/DA/`)의 가이드 문서 작성·갱신 시 아래 규칙을 적용한다.

### 6.1. 메타데이터·변경이력 준용

. 각 관점 가이드 문서는 `공통섹션/메타데이터.md`·`공통섹션/변경이력.md` 의 구조를 준용한다.
. 상단 메타데이터 표와 하단 변경이력 4컬럼 표는 기획 공통섹션과 동일 양식을 사용한다.
. 관점 고유 필드(예: 개발 가이드의 "적용 언어·프레임워크", QA 가이드의 "적용 테스트 레벨")는 추가 허용한다.

### 6.2. 파일명 규칙

. 관점 가이드는 `Context/[관점]/[주제].md` 형식을 사용한다 (예: `개발/네이밍규칙.md`).
. Docs/ 에 산출되는 성과물이 아닌 내부 참조 문서이므로 버전 접미어(`_v0.1`)를 파일명에 포함하지 않는다.
. 파일명 버전 구분자 규칙(CLAUDE.md §6.2)은 Docs/ 성과물 대상이며 Context/ 내부 가이드에는 적용하지 않는다.

### 6.3. 문체·형식

. `기획/가이드/문체규칙.md` 의 종결 어미·어휘·주어 명시 규칙을 동일하게 적용한다.
. CLAUDE.md §1 형식 선택 기준(A·B)도 동일하게 적용한다.
. 비교·매트릭스성 정보는 반드시 형식 B(테이블)로 표현한다.

### 6.4. 신규 관점 문서 추가 절차

1. 대상 관점 폴더를 선택한다(개발/디자인/QA/DA).
2. 주제를 단일 책임 원칙으로 분할한다. 하나의 파일은 하나의 주제만 다룬다.
3. 파일 생성 후 해당 관점의 `INDEX.md` 목록에 1줄 설명을 추가한다.
4. 상위 `Context/INDEX.md` 의 관점별 표에도 1줄 설명을 추가한다.
5. 규칙 충돌 여지가 있다면 `Context/INDEX.md` SSOT 매트릭스에 신규 행을 추가한다.

### 6.5. 자가점검 적용

. 신규 관점 문서 작성·갱신 시에도 CLAUDE.md §7 자가점검 워크플로가 자동 적용된다.
. 상세 구현은 `Context/자가점검_워크플로.md` 를 참조한다.
. 관점별 특화 점검(코드 린트·DB 스키마 검증·테스트 커버리지 등)은 동 문서 §3 M18 이후의 확장 방법 풀을 따른다.

단, 관점 가이드의 버전업은 Patch/Minor/Major 구분 없이 단일 버전 체계(v1.0.0 → v1.0.1 → v1.1.0)를 사용한다.
※ 관점 간 경계 파일(예: `개발/테스트작성.md` ↔ `QA/테스트레벨.md`)은 상호 링크를 반드시 포함한다.

## 1. 서비스 개요

. 서비스 가칭 : Multi-Model Product Dev-Agent (가칭)
. 서비스 정의 : 제품 기획, 시스템 설계, UI/UX 디자인, 소스 코드 구현 및 검증의 전 과정을 복수의 특화 AI 모델(Claude, Gemini, Llama, Midjourney)을 통해 통합 수행하는 일렉트론 기반의 지능형 개발 에이전트 시스템이다.
. 핵심 컨셉 : 하이브리드 에이전틱 워크플로우(Hybrid Agentic Workflow). PC 애플리케이션의 강력한 제어권과 구글 스페이스의 협업 편의성을 결합하여 제품 개발의 전 주기를 관리한다.

---

## 2. 기획 배경 및 상세 목적

. 파편화된 도구의 한계 : 기존 AI 툴들은 기획 따로, 코드 따로 작동하여 기획서의 변경 사항이 코드에 실시간으로 반영되지 않는 컨텍스트 단절 문제가 발생한다.
. 데이터 무결성 결여 : 수동으로 복사-붙여넣기를 반복하는 과정에서 휴먼 에러가 발생하며, 이는 곧 시스템 설계의 결함으로 이어진다.
. 지식 자산화 부재 : 개발 과정에서의 의사결정 이유(Context)는 사라지고 최종 결과물만 남게 되어, 유지보수 시 과거의 설계 의도를 파악하기 어렵다.
. 최종 목적 : 기획 의도와 구현물 사이의 정합성을 100% 유지하고, 모든 개발 과정을 추적 가능한 지식 위키로 자산화하여 제품 개발의 리드 타임을 혁신적으로 단축하는 데 있다.

---

## 3. 타겟 사용자 및 상세 페르소나

. 주 사용자 : 기술적 의사결정이 가능한 제품 매니저(PM) 및 풀스택 개발자

### 페르소나 1 — 1인 창업자

. 비즈니스 아이디어는 확실하나 디자인과 초기 아키텍처 설계에 투입할 자원이 부족하다.
. 에이전트를 파트너 삼아 MVP(Minimum Viable Product)를 1주일 이내에 개발하고자 하는 User이다.

### 페르소나 2 — 소규모 팀 리더

. 팀원 간의 컨텍스트를 동기화하고, 프로젝트의 모든 변경 이력을 자동으로 문서화하여 기술 부채를 최소화하려는 관리자이다.

## 4. 핵심 가치 (Core Value) 상세 정의

| 핵심 가치 | 정의 |
|---|---|
| 논리적 일관성 (Logical Consistency) | 기획 단계에서 정의된 데이터 모델과 비즈니스 로직이 설계와 구현 단계를 거치며 변질되지 않도록 System이 강제한다. |
| 실행 연속성 (Seamless Execution) | 환경 설정, 의존성 설치, 코드 생성, 빌드, 테스트로 이어지는 과정에서 User의 개입을 최소화하고 중단 없는 흐름을 제공한다. |
| 지식 영속성 (Knowledge Permanence) | 단순한 결과물이 아닌, 결과물이 도출되기까지의 사고 과정(Chain of Thought)을 모두 저장하여 프로젝트의 진정한 소유권을 User에게 부여한다. |

## 5. 상세 기능 및 모델 오케스트레이션 설계

### 5.1. 기획 엔진 (Claude 3.5 Sonnet / Gemini 1.5 Pro)

. 역할 : 요구사항 명세서(PRD), 유즈케이스 시나리오, 사용자 여정 지도(User Journey Map) 생성
. 특화 기능 : Gemini의 긴 컨텍스트 윈도우를 활용하여 대규모 기존 레거시 문서를 분석하고 신규 요구사항과의 충돌 여부를 검증한다.

### 5.2. 설계 엔진 (Claude 3.5 Sonnet)

. 역할 : 데이터베이스 스키마(ERD), API 엔드포인트 설계, 인프라 구성도(IaC) 초안 작성
. 특화 기능 : 기획서의 명사를 추출하여 DB 테이블을 설계하고, 동사를 추출하여 API 핸들러 로직을 설계한다.

### 5.3. 시각 엔진 (Midjourney / Gemini 1.5 Flash)

. 역할 : 서비스 브랜드 가이드라인 생성, 와이어프레임 및 고해상도 UI 목업 생성
. 브릿지 로직 : Midjourney의 이미지 결과물을 Gemini Flash가 멀티모달 분석하여 Tailwind CSS 클래스 및 React 컴포넌트 구조로 치환한다.

### 5.4. 구현 엔진 (Llama 3 / Claude 3.5 Sonnet)

. 역할 : 폴더 구조 자동 생성, 보일러플레이트 작성, 실제 비즈니스 로직 코딩 및 단위 테스트 작성
. 특화 기능 : 로컬 파일 시스템에 직접 접근하여 파일을 생성하고, 변경된 부분만 정밀하게 패치(Patch)한다.

### 5.5. 오케스트레이터 로직

. 분류 (Classification) : System이 사용자 입력을 분석하여 기획, 디자인, 구현 작업으로 분류한다.
. 문맥 주입 (Context Injection) : System이 이전 단계의 산출물을 다음 단계 모델의 프롬프트에 자동으로 주입하여 연속성을 유지한다.
. 결과 검증 (Self-Criticism) : System이 생성된 산출물을 별도의 검증 모델로 비즈니스 요구사항과 대조하여 통과 여부를 결정한다.

## 6. 프로젝트 하이라키 및 데이터 상속 규정 상세

### Layer 1. 루트 (Root)

. 데이터 정의 : 전사 표준 폰트, 컬러 팔레트, 로깅 프로토콜, 보안 취약점 체크리스트
. 상속 방식 : 하위 레이어에서 자동으로 포함(Include)되며, 전역 상수를 통해 접근한다.

### Layer 2. 부모 (Parent)

. 데이터 정의 : 공통 사용자 인증 모듈, 통합 알림 시스템, 핵심 도메인 엔티티(User, Product 등)
. 상속 방식 : 부모의 데이터 모델은 하위에서 참조 가능하며, 확장(Extend)을 통해 추가 필드를 정의할 수 있다.

단, 부모 필드의 삭제나 타입 변경은 System이 거부한다.

### Layer 3. 자식 (Child)

. 데이터 정의 : 특정 마이크로 서비스 로직, 전용 테마 CSS, 개별 기능 테스트 코드
. 독립성 : 자식 간의 데이터 침범은 물리적으로 격리되며, 오직 동일 부모를 공유하는 형제 프로젝트 간의 API 통신만 허용된다.

## 7. 실행 환경 및 인터페이스 상세 인터랙션

### 7.1. PC 애플리케이션 (The Brain)

. 핵심 모듈 : Node.js 런타임 기반의 파일 시스템 제어 모듈, 실시간 로그 스트리밍 뷰어, 프로젝트 대시보드
. 로컬 연동 : 사용자 환경의 IDE(VS Code 등)와 소켓 통신을 통해 실시간 코드 동기화를 지원한다.

### 7.2. 구글 스페이스 (The Communicator)

. 알림 프로토콜 : 빌드 성공/실패, AI 모델의 주요 질문사항, 프로젝트 마일스톤 달성 보고
. 리모트 컨트롤 : 외부에서 구글 스페이스 채팅을 통해 간단한 코드 수정 지시나 문서 업데이트 명령을 전달하면 PC 앱이 이를 수신하여 실행한다.

## 8. 데이터 관리 및 하이퍼 스냅샷 롤백 정책

. 이벤트 소싱 (Event Sourcing) : System이 모든 상태 변화를 순차적인 이벤트 로그로 저장한다.

### 데이터 저장 구조

| 대상 | 저장소 | 관리 단위 |
|---|---|---|
| 대화 및 사고 로그 | NoSQL DB | 타임스탬프 단위 |
| 코드 및 에셋 | 로컬 Git 저장소 | 커밋 단위 |
| 스냅샷 정보 | Vector DB | 전체 컨텍스트(모델 상태, 변수 값 등) 인덱싱 |

### 롤백 워크플로우

| 단계 | 트리거 | Client 동작 | Server 동작 | DB / 외부 연동 |
|---|---|---|---|---|
| 1. 복구 지점 선택 | User가 PC 앱의 타임라인 인터페이스에서 특정 복구 지점을 선택한다 | Client가 타임라인 UI에서 선택된 스냅샷을 하이라이트한다 | - | - |
| 2. 파일 시스템 롤백 | User가 롤백 확인 버튼을 클릭한다 | Client가 롤백 진행 상태를 출력한다 | Server가 로컬 파일 시스템을 해당 시점의 Git 커밋으로 되돌린다 | Git: `git reset --hard [commit_hash]` |
| 3. 컨텍스트 복구 | 파일 시스템 롤백 완료 시 | Client가 복구 완료 메시지를 출력한다 | Server가 로그 DB와 스냅샷 데이터를 복구한다 | NoSQL DB: 해당 타임스탬프 이후 이벤트 소프트 삭제, Vector DB: 스냅샷 컨텍스트 로드 |

## 9. 지식 위키 및 시맨틱 지식 인출 시스템

### 위키 생성 로직

. System이 확정된 요구사항, 아키텍처 결정 사항(ADR), API 명세서를 자동으로 Markdown 문서화하여 내부 위키 저장소에 저장한다.

### 지식 Taxonomy (분류 체계)

| 분류 | 내용 | 예시 |
|---|---|---|
| 의사결정 이력 | 특정 기술 선택의 이유와 맥락 | "왜 이 라이브러리를 사용했는가?" |
| 기술 명세 | 서비스의 구조와 인터페이스 정의 | API 엔드포인트 목록, ERD |
| 트러블슈팅 | 발생했던 에러와 AI가 제안한 해결책 | 에러 로그 + 수정 코드 + 기획 의도 |

### 시맨틱 검색 엔진

. System이 사용자 질문의 의도를 파악하여 관련성이 높은 위키 페이지와 대화 로그를 추천한다.
. 예시 : "결제 실패 사례"를 검색하면 System이 당시의 코드, 로그, 기획 의도를 한데 묶어 제공한다.

## 10. 보안 및 운영 가이드라인

. 보안 프로토콜 : System이 로컬 데이터를 AES-256 방식으로 암호화하며, AI 모델 전송 전 개인정보(PII) 자동 마스킹 기능을 가동한다.
. 비용 최적화 : System이 작업의 중요도에 따라 고성능 모델(Claude Opus)과 경량 모델(Gemini Flash)을 유동적으로 할당하여 API 비용을 절감한다.
. 에러 처리 : 모델 응답 실패 시 System이 지수 백오프(Exponential Backoff) 기반의 재시도 로직을 적용하고, 실패 지속 시 구글 스페이스를 통해 즉시 Admin에게 긴급 알림을 전송한다.

Figma 디자인을 개발 팀으로 전달하는 핸드오프 절차와 검수 체크리스트를 정의한다. Designer는 개발 착수 전 본 문서의 기준을 충족하는 상태로 파일을 정리한다.

---

## 1. 핸드오프 대상 산출물

| 산출물 | 형식 | 제공 경로 |
|---|---|---|
| 화면 디자인 | Figma Frame URL | Figma 공유 링크 (Dev Mode 권한) |
| 컴포넌트 라이브러리 | Figma Library | 공유 Team Library |
| 디자인 토큰 | JSON (Tokens Studio Export) | GitHub `tokens/` 디렉토리 |
| 아이콘·이미지 | SVG·PNG·WebP | GitHub `assets/` 디렉토리 |
| 프로토타입 | Figma Prototype | Figma 공유 링크 |
| 디자인 명세서 | Markdown | `Docs/디자인명세서_v*.md` |

## 2. Figma Dev Mode 활용

. Designer는 핸드오프 대상 Frame을 `Ready for dev` 상태로 마킹한다.
. Dev Mode에서 Measurement·CSS Inspector·Token Reference가 모두 정상 노출되는지 검증한다.
. Branch 사용 시 핸드오프 대상은 반드시 `main` Branch에 Merge한 상태여야 한다.
. Code Connect가 설정된 컴포넌트는 Dev Mode에서 프로젝트 코드 스니펫이 자동 매핑된다.

## 3. 토큰 Export 규칙

Designer는 Figma Variables를 Tokens Studio 플러그인을 통해 W3C Design Tokens 포맷으로 export한다.

```json
{
  "color": {
    "primary": {
      "500": { "value": "#2E75B6", "type": "color" }
    }
  },
  "spacing": {
    "md": { "value": "16px", "type": "dimension" }
  }
}
```

| 카테고리 | 파일명 | 소스 |
|---|---|---|
| Color | `tokens/color.json` | `컬러시스템.md` |
| Typography | `tokens/typography.json` | `타이포그래피.md` |
| Spacing | `tokens/spacing.json` | `간격시스템.md` |
| Radius | `tokens/radius.json` | - |
| Shadow | `tokens/shadow.json` | - |

## 4. 네이밍 매핑

Figma 컴포넌트와 코드 컴포넌트 간 매핑은 `컴포넌트네이밍.md §1~§4` 규칙을 따른다. 개발 팀은 Code Connect 매핑 파일(`.figma.ts`)을 작성하여 자동 매핑을 구성한다.

```typescript
// Button.figma.ts 예시
import figma from "@figma/code-connect"
import { Button } from "./Button"

figma.connect(Button, "https://figma.com/...", {
  props: {
    variant: figma.enum("variant", { primary: "primary", secondary: "secondary" }),
    size: figma.enum("size", { sm: "sm", md: "md", lg: "lg" }),
    disabled: figma.enum("state", { disabled: true }),
  },
  example: ({ variant, size, disabled }) => (
    <Button variant={variant} size={size} disabled={disabled}>Label</Button>
  ),
})
```

## 5. 핸드오프 전 검수 체크리스트

Designer는 아래 항목을 모두 통과한 후 핸드오프를 진행한다.

| 분류 | 항목 | 기준 |
|---|---|---|
| 기본 | 모든 Layer가 의미 기반 명칭으로 정리됨 | `컴포넌트네이밍.md §5` |
| 기본 | Frame은 Auto-layout 적용, 절대 위치 최소화 | - |
| 토큰 | 모든 Color·Typography·Spacing이 Variable로 바인딩됨 | 하드코딩 Hex·px 0건 |
| 토큰 | Effect(Shadow·Blur)가 Library Style로 연결됨 | - |
| 상태 | 인터랙션 컴포넌트의 7상태가 모두 제작됨 | `상태표현.md §1` |
| 반응형 | Mobile·Tablet·Desktop 3종 Frame 제공 | `반응형규칙.md §2` |
| 접근성 | 텍스트·아이콘 Button 대비 4.5:1 이상 | `접근성.md §2` |
| 접근성 | Focus 상태 시안 포함 | `접근성.md §3` |
| 자산 | 아이콘 SVG export 설정 완료 | `아이콘규칙.md §5` |
| 명세 | Dev Note·주석이 핵심 동작 Frame에 부착됨 | - |
| Prototype | 주요 Flow의 Interactive Prototype 제공 | - |

## 6. 커뮤니케이션 Flow

| 단계 | 주체 | 동작 | 산출 |
|---|---|---|---|
| 1. 디자인 리뷰 | Designer → PM | Figma 링크 공유 및 시안 리뷰 진행 | 리뷰 미팅록 |
| 2. 디자인 확정 | PM·Designer | 시안 승인 및 `Ready for dev` 마킹 | Figma 상태 변경 |
| 3. 핸드오프 킥오프 | Designer → Developer | 본 체크리스트 전달 및 Q&A | Kickoff 미팅 |
| 4. 개발 진행 | Developer | Dev Mode·Code Connect 활용 구현 | Pull Request |
| 5. 디자인 QA | Designer | 구현 결과를 시안과 대조 검수 | QA 이슈 리스트 |
| 6. 수정·재QA | Developer·Designer | 이슈 처리 및 재검수 | 최종 승인 |

단, QA 라운드는 최대 2회로 제한한다. 3회차 이상 필요 시 PM에게 에스컬레이션한다.

## 7. 디자인 QA 기준

Designer는 구현된 화면을 Figma 시안과 비교하여 아래 항목을 검수한다.

. Pixel-perfect : 주요 Breakpoint에서 ±1px 이내 일치
. 컬러·타이포·간격 : 토큰 값 정확히 반영
. 상태 전환 : hover·active·focus·disabled 모두 시안과 동일
. 반응형 : 브레이크포인트 전환이 시안과 일치
. 접근성 : 대비·Focus·ARIA 속성 적용 여부

※ QA 이슈는 Figma Comment 또는 Jira 이슈로 기록하며 Screenshot을 첨부한다.

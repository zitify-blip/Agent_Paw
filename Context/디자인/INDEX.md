## 2. 사용 순서

신규 Designer가 프로젝트에 합류할 경우 아래 순서로 학습한다.

. 1단계 — 기반 : `컬러시스템.md` → `타이포그래피.md` → `간격시스템.md` → `레이아웃그리드.md`
. 2단계 — 적용 : `아이콘규칙.md` → `상태표현.md` → `반응형규칙.md`
. 3단계 — 협업 : `컴포넌트네이밍.md` → `접근성.md` → `핸드오프.md`

## 4. 관점 간 경계

타 관점과 접점이 있는 파일은 상대 문서를 명시적으로 참조한다.

| 디자인 측 파일 | 상대 관점 파일 | 경계 내용 |
|---|---|---|
| `상태표현.md` | `개발/에러처리.md` | error/loading 상태 UI는 디자인, 실제 에러 처리·예외 클래스는 개발 |
| `핸드오프.md` | `개발/PR규칙.md`·`개발/코드리뷰.md` | 디자인 완료 → 개발 수령 전환점 |
| `접근성.md` | `QA/릴리즈검증.md` | WCAG 준수 검증은 QA 릴리즈 검증 체크리스트에 포함 |
| `컴포넌트네이밍.md` | `개발/네이밍규칙.md` | Figma 컴포넌트명과 코드 컴포넌트명의 일관성 유지 |

## 5. 자가점검 적용

. 디자인 산출물 변경 시 `Context/자가점검_워크플로.md` 의 M01·M02·M17·M21(디자인 토큰·접근성 일관성) 을 필수로 적용한다.
. 2개 이상 관점 동시 변경 시 M22(관점 간 경계) 를 1회 이상 포함한다.
